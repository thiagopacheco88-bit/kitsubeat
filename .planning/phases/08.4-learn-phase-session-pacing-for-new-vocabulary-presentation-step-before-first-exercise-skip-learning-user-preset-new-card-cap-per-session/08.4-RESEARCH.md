## Summary

Phase 08.4 inserts a just-in-time learn card before a user's first question for any new or relearning vocabulary word, with a global opt-out (`skip_learning`) and a per-session cap on how many new words can be introduced.

The exercise engine today is a pure client-side question loop: `buildQuestions()` (in `src/lib/exercises/generator.ts`) generates questions from the full lesson vocab at session start; `ExerciseSession.tsx` renders them sequentially by index; and `src/stores/exerciseSession.ts` (Zustand + localStorage) persists the entire question array and current index. JIT insertion must hook into `ExerciseSession`'s render loop — **not** into `buildQuestions()` — because the learn card is UI-only (it doesn't produce a question record) and must slot between `questions[currentIndex - 1]` and `questions[currentIndex]`.

No `users` table exists yet. `skip_learning` and `new_card_cap` need a new `users` table (or can be stored on `subscriptions` as user-level columns — see below). A Drizzle migration `0005_user_prefs.sql` adds this. No settings surface exists; the minimum viable surface is a simple toggle + number input inside the existing `ExerciseTab` config screen (before session starts) or a new `/profile` route — the profile route is recommended so it doesn't clutter the exercise UI.

**Primary recommendation:** Hook learn card into `ExerciseSession.tsx` as a conditional render before `<QuestionCard>` when `showLearnCard(state)` is true; track "shown" state in the Zustand store alongside the existing `revealedQuestionIds` map; apply cap filtering at session-start time in `ExerciseTab.handleStart` immediately after the vocab-tiers batch fetch, before `buildQuestions()`.

## Area 1: Exercise Engine + Session Store Integration Points

### Where questions are selected and rendered

**File:** `src/lib/exercises/generator.ts` — `buildQuestions(lesson, mode, jlptPool): Question[]`
- Pure function. Generates all possible questions from `lesson.vocabulary`, shuffles, slices to count.
- Does NOT know about FSRS state — tiers are fetched separately after session start.
- **No FSRS state filtering happens here today.** The cap must be applied in `ExerciseTab.handleStart` before `buildQuestions()` (or by filtering the lesson vocab passed in).

**File:** `src/app/songs/[slug]/components/ExerciseTab.tsx` — `handleStart(mode)`
- Fetches JLPT pool, calls `buildQuestions()`, calls `startSession()`, then batch-fetches FSRS tiers via `GET /api/exercises/vocab-tiers`.
- **Cap insertion point:** After tiers are fetched (line ~143), compare each `vocabItemId`'s state (state=0 = New, state=3 = Relearning) against the cap. Exclude words beyond cap from the lesson vocab before calling `buildQuestions()`. This is the cleanest place.

**File:** `src/app/songs/[slug]/components/ExerciseSession.tsx`
- Renders `questions[currentIndex]` in a fade transition loop.
- Shows `SessionSummary` when `currentIndex >= total`.
- **JIT learn card insertion point:** Before rendering `<QuestionCard key={current.id}>`, check if the current question is the first question for a new/relearning word AND the learn card hasn't been shown yet → render `<LearnCard>` instead. On any user interaction, set a flag in the store and re-render `<QuestionCard>` for the same index (do NOT advance `currentIndex`).
- Progress bar and "Question N / M" counter: these derive from `currentIndex` and `questions.length`. The learn card does NOT advance `currentIndex`, so the counter stays stable during the card — **this is correct UX** (the learn card isn't a "question").

**File:** `src/stores/exerciseSession.ts` — Zustand + localStorage persist
- Current shape: `{ songVersionId, questions, currentIndex, answers, startedAt, mode, tiers, revealedQuestionIds, moreAccordionOpen }`
- All state persists via localStorage (survives reload). `_hasHydrated` is runtime-only and excluded from persistence.
- **Session-store extensions needed:**
  - `learnedVocabIds: Record<string, true>` — tracks which `vocabItemId`s have had their learn card shown this session. Persisted so reload doesn't re-show the card mid-session.
  - `introducedNewVocabIds: Record<string, true>` — tracks which new/relearning words have been introduced this session (for cap accounting across question re-encounters of the same word). Also persisted.
- **Actions needed:**
  - `markLearnCardShown(vocabItemId: string)` — sets `learnedVocabIds[vocabItemId] = true`
  - `markVocabIntroduced(vocabItemId: string)` — sets `introducedNewVocabIds[vocabItemId] = true`
- **Reset in `startSession()`:** Both new fields reset to `{}` on session start (same as `tiers` and `revealedQuestionIds`).

### How new vs. review words are distinguished

Today: `ExerciseTab` fetches `GET /api/exercises/vocab-tiers?ids=...&userId=...` after session start → returns `{ tiers: { [vocabItemId]: 1 | 2 | 3 } }`.

The tiers API (`src/app/api/exercises/vocab-tiers/route.ts`) queries `user_vocab_mastery.state` and calls `tierFor(state)`:
- state=0 (New, no mastery row default) → Tier 1
- state=1 (Learning) → Tier 2
- state=2 (Review) → Tier 3
- state=3 (Relearning) → Tier 2

**Problem:** The tiers endpoint conflates New (state=0) and Relearning (state=3) into the same Tier values (1 and 2 respectively), but the learn card needs to distinguish them. Options:
1. **Extend the tiers API** to also return `state` alongside `tier` — add `states: { [vocabItemId]: 0|1|2|3 }` to the response. Learn card triggers on state=0 OR state=3.
2. **Add a separate batch endpoint** `GET /api/exercises/vocab-states?ids=...&userId=...` that returns raw FSRS states.
3. **Inline state check in ExerciseTab** by reading the raw tiers and re-querying — over-complex.

**Recommendation:** Option 1 — extend `GET /api/exercises/vocab-tiers` response to include a `states` map. Low coupling change, single request. Store `states` in Zustand alongside `tiers`.

## Area 5: Users Table / Settings Surface

### Users table

**No `users` table exists today.** The schema has `subscriptions` (with `user_id`), `user_vocab_mastery`, `user_exercise_log`, `user_song_progress` — all keyed by `user_id` text. Auth is not yet integrated (Clerk TODOs throughout).

**Decision:** Add a `users` table as the home for user preferences. This is the right long-term design: `subscriptions` is provider-agnostic billing data, not the right place for UX prefs.

```sql
-- drizzle/0005_user_prefs.sql
CREATE TABLE IF NOT EXISTS "users" (
  "id" text PRIMARY KEY NOT NULL,  -- Clerk user_id (text, not uuid)
  "skip_learning" boolean DEFAULT false NOT NULL,  -- false = show cards (ON by default)
  "new_card_cap" integer DEFAULT 5 NOT NULL,       -- per-session new word cap
  "created_at" timestamp with time zone DEFAULT now() NOT NULL,
  "updated_at" timestamp with time zone DEFAULT now() NOT NULL
);
```

Note: `skip_learning DEFAULT false` means "do NOT skip" = cards show. This maps correctly to the CONTEXT "default ON (cards show)".

## Area 6: TTS Options for Japanese

No TTS infrastructure exists in the codebase today. The only audio pipeline is wavesurfer.js (`@wavesurfer/react` in dependencies) used for song audio waveform display in the timing editor — unrelated to TTS.

### Option A: Web Speech API (`SpeechSynthesis`) — RECOMMENDED

- **Cost:** Free (browser-native)
- **Latency:** Near-zero (no network round-trip)
- **Japanese support:** Available on Chrome, Safari, Edge with `ja-JP` voice. Firefox support is inconsistent. On mobile: iOS Safari has good Japanese TTS; Android Chrome generally works.
- **Quality:** Acceptable for short vocabulary words (single words read clearly). Unnatural for sentences but learn cards show single vocab items.
- **Implementation:** `window.speechSynthesis.speak(new SpeechSynthesisUtterance(text))` with `utterance.lang = 'ja-JP'`
- **Caching:** None needed — synthesis is immediate
- **Pitfall:** `speechSynthesis.getVoices()` may return empty array on first call; must listen for `voiceschanged` event. Some browsers pause synthesis if the tab loses focus.
- **Zero dependencies:** No package to add, no API key, no cost.

**Confidence: MEDIUM** — Japanese voice availability on all browsers is not 100% guaranteed; need a graceful fallback (hide speaker icon if no Japanese voice found).

### Option B: Google Cloud TTS

- **Cost:** ~$4/1M characters (generous free tier: 1M chars/month standard, 1M chars/month WaveNet)
- **Latency:** ~200–500ms network round-trip; requires server-side proxy to hide API key
- **Japanese quality:** Excellent (WaveNet voices). Best option for quality.
- **Caching strategy:** Cache audio blobs in Vercel KV or edge cache keyed by `sha256(text + voice)`. Most vocabulary words are ≤ 10 chars; cache hit rate would be high after first user.
- **Implementation complexity:** Server-side API route + client fetch + audio blob URL
- **Cost for phase:** Likely under free tier for current user volume

### Option C: Azure Cognitive Services Speech

- Similar to Google Cloud TTS in cost and quality. Requires Azure account.

### Recommendation

**Use Web Speech API (Option A) for v1.** Rationale:
1. Zero cost, zero dependencies, zero backend
2. Japanese vocab words are short — TTS quality is acceptable
3. Can be swapped out for Google Cloud TTS in a later phase if quality complaints arise
4. Implementation is < 20 lines
5. Graceful degradation: detect `'speechSynthesis' in window && voices.some(v => v.lang.startsWith('ja'))` before showing the speaker icon; hide it if not available

**Implementation pattern:**
```ts
// src/lib/tts.ts
export function speakJapanese(text: string): void {
  if (typeof window === 'undefined' || !('speechSynthesis' in window)) return;
  const voices = window.speechSynthesis.getVoices();
  const jaVoice = voices.find(v => v.lang.startsWith('ja'));
  const utt = new SpeechSynthesisUtterance(text);
  utt.lang = 'ja-JP';
  if (jaVoice) utt.voice = jaVoice;
  window.speechSynthesis.cancel(); // stop any current speech
  window.speechSynthesis.speak(utt);
}
```

**Effective vocab-with-ID estimate:**

### Cap recommendation

**Target: 2–4 sessions to complete a song for a beginner.**

If all words are new and cap = N per session:
- Song with 20 new words (effective minimum): 20/N sessions
- Song with 40 new words (p25): 40/N sessions  
- Song with 60 new words (median): 60/N sessions

For 2–4 session progression on a **median song** (60 words): cap = 15–30.
For 2–4 session progression on a **p25 song** (40 words): cap = 10–20.

**Recommended default cap: 10 new words per session.**

Rationale:
- 10 new words + their 4 exercise types = up to 40 questions per session (the "full" mode cap)
- At cap=10: median song completes in ~4–6 sessions; a smaller song (20 words) in 2 sessions
- This is conservative but protects beginners from overwhelm — the premium unlock lets motivated users raise it
- Aligns with Anki's default of 10–20 new cards/day (per-session equivalent for this app)
- ExerciseTab already shows session counts: `shortCount = Math.min(10, vocabCount * 4)` and `fullCount = Math.min(40, vocabCount * 4)` — a cap of 10 new words means ~40 questions max in full mode, which fits within current design

**Premium cap ceiling:** Allow premium users to set up to 30 (3× default). Store as int; validate server-side.

## Area 8: Architecture Patterns

### Pattern 1: JIT Learn Card Insertion in ExerciseSession

**What:** `ExerciseSession` renders a `LearnCard` instead of `QuestionCard` when the current question is the first encounter with a new/relearning word and the learn card hasn't been shown yet.

**State machine:**
```
For each question at currentIndex:
  IF vocab is new/relearning AND learnedVocabIds[vocabItemId] is not set
    → render <LearnCard vocab={...} onDismiss={markLearnCardShown(vocabItemId)} />
    → do NOT advance currentIndex
  ELSE
    → render <QuestionCard ...> (existing behavior)
```

**Dismiss action:** Any tap on the card body → `markLearnCardShown(vocabItemId)` → component re-renders → now renders `<QuestionCard>` for the same `currentIndex`.

**skipLearning bypass:** If `user.skip_learning === true`, skip the learn card check entirely — go straight to `<QuestionCard>`. The `skip_learning` value must be fetched server-side and passed as a prop to `ExerciseSession` (or via a context).

### Pattern 2: Cap Filtering Before buildQuestions

**What:** After fetching vocab tiers/states in `ExerciseTab.handleStart`, filter the lesson vocab to exclude new/relearning words beyond the cap.

```ts
// After fetching states (extended vocab-tiers response):
const newVocabIds = lesson.vocabulary
  .filter(v => v.vocab_item_id && (stateMap[v.vocab_item_id] === 0 || stateMap[v.vocab_item_id] === 3))
  .map(v => v.vocab_item_id!);

// Apply cap (server-enforced: cap value comes from checkCapAccess):
const cappedNewIds = new Set(newVocabIds.slice(0, userCap));

// Build filtered lesson vocab:
const filteredVocab = lesson.vocabulary.filter(v => {
  if (!v.vocab_item_id) return false;
  const state = stateMap[v.vocab_item_id] ?? 0;
  // Review and Learning words always included; new/relearning only if within cap
  if (state === 0 || state === 3) return cappedNewIds.has(v.vocab_item_id);
  return true;
});

const questions = buildQuestions({ ...lesson, vocabulary: filteredVocab }, mode, jlptPool);
```

**Cap enforcement:** The `userCap` value must come from the server (via a new server action or extended session-start endpoint) to prevent client-side tampering. Free users must not be able to override the default cap by editing localStorage.

### Pattern 3: New Server Action for User Prefs

```ts
// src/app/actions/userPrefs.ts
"use server";

export async function getUserPrefs(userId: string): Promise<{ skipLearning: boolean, newCardCap: number }>;
export async function updateUserPrefs(userId: string, prefs: Partial<{ skipLearning: boolean, newCardCap: number }>): Promise<void>;
export async function getEffectiveCap(userId: string): Promise<number>; // enforces premium ceiling
```

`getEffectiveCap` is the premium gate: checks `subscriptions` table for active premium plan; if premium, returns `users.new_card_cap` (capped at 30); if free, returns `DEFAULT_CAP` (10) regardless of stored value.

## Area 9: Common Pitfalls

### Pitfall 1: Progress counter mismatch during learn card

**What goes wrong:** "Question N / M" counter in `ExerciseSession.tsx` reads `currentIndex + 1` and `questions.length`. If learn card is shown at `currentIndex=3`, the counter shows "Question 4 / N" — which is correct (the user is on question 4, they just haven't seen it yet). This is fine.

**Potential issue:** If the progress bar advances visually while the learn card is showing, users may think the learn card "counts" as a question. Since `currentIndex` is not advanced during learn card display, the progress bar stays at the correct position.

**How to avoid:** No change needed to progress bar logic. The learn card is transparent to `currentIndex`.

### Pitfall 2: Session store persistence and "already shown" tracking

**What goes wrong:** If the user closes the browser mid-session and resumes, `learnedVocabIds` must persist to avoid re-showing learn cards for words already taught. The existing Zustand store uses `localStorage` via `persist` middleware.

**How to avoid:** Include `learnedVocabIds` and `introducedNewVocabIds` in the persisted slice (do NOT add them to the `partialize` exclusion list — unlike `_hasHydrated`, these must survive reloads). Verify the `partialize` function in `exerciseSession.ts` (line 150) only excludes `_hasHydrated`.

### Pitfall 3: skip_learning + new user (no mastery data)

**What goes wrong:** If `skip_learning=true` for a user with no mastery data, the system should still present exercises normally with FeedbackPanel teaching post-answer. The filter logic must not break when there are no mastery rows.

**How to avoid:** When `skip_learning=true`, skip learn card rendering entirely but apply the cap filter as normal. New words (state=0 / no row) still count against the cap and are gated by it. FeedbackPanel already handles the teaching-after-answer flow with no changes needed.

### Pitfall 4: Cap value downgrade reconciliation

**What goes wrong:** A premium user sets cap=25, downgrades to free. Their `users.new_card_cap` column still holds 25, but `getEffectiveCap()` must return 10.

**How to avoid:** `getEffectiveCap()` always derives the enforced cap from `subscriptions.plan + subscriptions.status`, never trusts `users.new_card_cap` alone. The stored cap is only used when the user is verified premium.

### Pitfall 5: Web Speech API `getVoices()` async initialization

**What goes wrong:** `speechSynthesis.getVoices()` returns `[]` on first call in Chromium browsers. The voices load asynchronously.

**How to avoid:**
```ts
function getJapaneseVoice(): SpeechSynthesisVoice | null {
  const voices = window.speechSynthesis.getVoices();
  return voices.find(v => v.lang.startsWith('ja')) ?? null;
}

// Init: listen for voiceschanged
window.speechSynthesis.addEventListener('voiceschanged', () => {
  // Re-check voice availability, show/hide speaker icon
});
```

Hide the speaker icon if no Japanese voice is found after `voiceschanged` fires. Do not show the icon and then break on tap.

### Pitfall 6: No users table — new table must be safe for unauthenticated users

**What goes wrong:** Auth is not yet integrated (Clerk TODOs throughout the codebase). The `users` table will have no rows until auth ships.

**How to avoid:** `getUserPrefs()` should handle missing rows gracefully — return defaults (`{ skipLearning: false, newCardCap: 10 }`) when no row exists for the userId. Since userId is currently a hardcoded placeholder (see TODOs in ExerciseTab, QuestionCard, ExerciseSession), this is fine for now.

### Pitfall 7: vocab-tiers API state extension — backward compat

**What goes wrong:** Extending `GET /api/exercises/vocab-tiers` to return `states` breaks callers that only destructure `{ tiers }`.

**How to avoid:** Add `states` as an additive field — existing callers continue to work. Update `ExerciseTab` to also destructure `states` from the response. The Zustand store gets a new `states: Record<string, 0|1|2|3>` field alongside `tiers`.

### Web Speech TTS (MEDIUM confidence — browser API, no training verification needed)

```ts
// src/lib/tts.ts
export function speakJapanese(text: string): void {
  if (typeof window === 'undefined' || !('speechSynthesis' in window)) return;
  const voices = window.speechSynthesis.getVoices();
  const jaVoice = voices.find(v => v.lang.startsWith('ja')) ?? null;
  const utt = new SpeechSynthesisUtterance(text);
  utt.lang = 'ja-JP';
  if (jaVoice) utt.voice = jaVoice;
  window.speechSynthesis.cancel();
  window.speechSynthesis.speak(utt);
}

export function hasJapaneseVoice(): boolean {
  if (typeof window === 'undefined' || !('speechSynthesis' in window)) return false;
  return window.speechSynthesis.getVoices().some(v => v.lang.startsWith('ja'));
}
```

## Open Questions

1. **Reveal animation micro-interaction** (Claude's discretion)
   - Options: (A) Inline expand (CSS `max-height` transition, simplest), (B) Overlay slide-up panel, (C) Flip card (CSS 3D transform)
   - **Recommendation: Inline expand** — consistent with the existing FeedbackPanel "More" accordion pattern. No new animation library needed. `max-height: 0 → auto` with `overflow-hidden` and `transition-all duration-300`.

2. **First tap vs. second tap to dismiss**
   - CONTEXT says "any tap/swipe advances." Does this mean:
     - (A) One tap on the card body dismisses it (mnemonic/kanji breakdown never visible), OR
     - (B) First tap reveals the hidden content; second tap advances
   - Reading CONTEXT: "mnemonic + kanji_breakdown behind a **single tap-to-reveal**" + "**any tap/swipe** advances" suggests (B): first tap reveals, second tap dismisses. This is confirmed by "any tap/swipe advances" applying to the **revealed state** (after first tap). **Recommendation: (B).**
   - If the user wants no-content-hidden experience they use `skip_learning`.

3. **Profile page or ExerciseTab settings?**
   - **Recommendation: `/profile` page** — keeps exercise flow uncluttered and sets up the settings surface for future prefs.

4. **`users` table primary key type**
   - Clerk user IDs are strings like `user_abc123`. Use `text PRIMARY KEY` not `uuid`. Matches the pattern of other `user_id text` columns in the schema.
