# Phase 08.4: Learn phase + session pacing for new vocabulary — Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Introduce a **just-in-time learn step** before a user's first exercise with a word they don't know, expose a **global `skip_learning` user preset** so power users can opt out, and enforce a **per-session new-card cap** so sessions don't overwhelm a user with too many unseen words at once. Scope is confined to the exercise flow — player, catalog, and cross-song features stay untouched.

Dependencies already shipped and relied upon:
- Phase 8: Exercise engine + session store
- Phase 08.2: FSRS per-vocab mastery (user_vocab_mastery with state column) and Tier derivation
- Phase 08.3: mnemonic + kanji_breakdown columns on vocabulary_items, FeedbackPanel "More" accordion, KanjiBreakdownSection component

</domain>

<decisions>
## Implementation Decisions

### Learn card content & flow
- **Content**: Full-stack breakdown on the card — kanji + furigana + romaji + meaning + part of speech — with **mnemonic + kanji_breakdown behind a single tap-to-reveal** (not progressive three-step reveal). Reuse the FeedbackPanel / KanjiBreakdownSection components from Phase 08.3 wherever possible.
- **Pacing**: **Interleaved just-in-time (JIT)** — no upfront batched "today's new words" screen. When the exercise engine is about to present a new word for the first time, the learn card appears inline immediately before that word's first question.
- **Acknowledgement**: **Just advance** — any tap/swipe dismisses the card. No explicit "Got it" button, no confidence rating. Minimal friction.
- **Audio**: **Tap-to-play TTS** via a speaker icon. No auto-play on card open. Audio is available but user-initiated.

### When the learn step triggers
- **Qualifying words**: **New + Relearning** — both words with no FSRS mastery row (brand new) and words the FSRS engine has dropped back into the relearning state (Tier 2 collapse after a failed review) get a learn card.
- **Re-show policy**: **Once per word per user, re-shown only after forgetting.** First time a qualifying word is encountered → card shows. Card does NOT show again on subsequent sessions while the word is progressing through FSRS normally. Card shows again only if FSRS drops the word back to relearning state.
- **Cap interaction**: **Learn card always shows for genuinely new words.** The cap is not a gate on the card itself — it's a gate on how many new words enter the session at all. Words excluded by the cap never reach the learn-card trigger point.
- **Cold-start**: **Normal cap applies from session 1.** No special bootstrap behavior for first-ever session; cap enforcement is uniform.

### skip_learning preset
- **Scope**: **Global user-level setting.** One toggle per user; applies everywhere. No per-song or per-session override (kept simple).
- **Default**: **ON** (learn cards show for new users). Power users opt out explicitly.
- **Effect when true**: **No learn cards are shown at all.** Exercises run exactly as they do today — new words are first seen in a question; the FeedbackPanel teaches after the answer. The new-card cap still applies (skip_learning does NOT disable pacing).
- **Storage**: **Column on the users table** — boolean `skip_learning` alongside the cap value (see below). No new preferences table; no JSONB settings blob. Explicit columns for now; migrate to a preferences surface later if the settings list grows.

### New-card cap behavior
- **Window**: **Per-session reset.** Cap is consumed as new words are introduced within a single exercise session and resets when the next session starts. No rolling 24-hour window, no midnight reset.
- **At cap**: **Remaining new words are deferred to the next session.** The current session continues using only the new words already introduced plus any review-eligible words from the song — session does NOT end early, does NOT show a "come back tomorrow" nudge.
- **User control — premium tiering**: **Free users are locked to the default cap. Premium users can raise it.** Enforcement lives at the data/access layer, same abstraction used by the Phase 08.1 exercise feature flags — never a UI-only gate.
- **Storage**: Cap value stored as an int column on the users table alongside `skip_learning`.

### Claude's Discretion
- **Default cap value** — research should survey typical anime song vocab density (~15–25 vocab entries per song per current catalog) and pick a default that yields a meaningful 2–4 session progression per song for a beginner. Planner locks the number.
- **Exact reveal animation / micro-interaction** for the tap-to-reveal on the card (inline expand vs. overlay vs. flip).
- **TTS provider / implementation** — which TTS engine, where audio URLs come from, caching policy. Not a product decision.
- **Session-store schema extensions** needed to track "which new words have been introduced this session" and "learn card shown for word X already" — internal data modeling, not user-facing.
- **Exact UI surface** for the skip_learning toggle + cap input (profile page? settings modal? exercise config screen?) — Claude picks based on what exists in the app today; if no settings surface exists, build the minimum viable one.
- **Premium gate wire-up** — how exactly cap-raising is gated (reuses the checkExerciseAccess pattern or a new gate) is an implementation detail the planner resolves.

</decisions>

<specifics>
## Specific Ideas

- "Interleaved JIT" is deliberate: user picked it over a batched Anki-style "today's new words" screen. The mental model is *"teach me right before you test me"*, not *"make me study a deck first"*.
- One-tap reveal (not progressive three-tap) keeps the card informative-by-default while avoiding an overwhelming wall of text on first glance.
- Premium tiering on cap adjustability is intentional — pacing is a premium-feel lever, not just a free-tier default. Ties into the existing premium-gate abstraction from Phase 08.1-01 / checkExerciseAccess.
- skip_learning only skips cards; it never disables the cap. Power users still get pacing; they just want the exercise-first flow.

</specifics>
