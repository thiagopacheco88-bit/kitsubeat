# Phase 08.1: End-to-End QA Suite - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Cross-cutting QA test infrastructure (Playwright + Node-side) that verifies the v1.0 player experience and the v2.0 Phase 8 exercise engine work end-to-end before Phase 9 (Kana Trainer) extends the codebase further. The suite is the gate between shipped Phase 8 work and any new feature that builds on top of it.

This phase delivers the QA infra + the first round of tests that exercise existing capabilities. It does NOT add new product features, new user-facing screens, or new exercise types.

</domain>

<decisions>
## Implementation Decisions

### Coverage scope
- **Equal depth across v1.0 player and Phase 8 exercise engine** — neither is smoke-only; both get deep coverage
- Player flows that MUST be tested:
  - Song loads + plays (song page opens, YouTube embed mounts, player ready, verse highlight follows playback ~250ms)
  - Interactive lesson (toggle furigana/romaji/translation, click token → dictionary popup with reading + meaning)
  - Grammar + vocab panels (GrammarSection + VocabularySection render with color-coding + grouping)
  - Verse timing sync across multiple verses including seek/skip
- Exercise flows that MUST be tested:
  - Full session completion (open exercise tab → 4 exercise types in sequence → immediate feedback each turn → summary screen)
  - Star earning + confetti (Star 1 at ≥80% vocab recognition, Star 2 at ≥80% fill-the-lyric, stars visible on SongCard + song page)
  - Resume mid-session (close browser mid-session, reopen, land exactly where left off — Zustand + DB persistence)
  - Progress + FSRS update (user_song_progress + user_vocab_mastery rows write/update, completion % reflects on SongCard)
- Regression guards (explicit tests required):
  - Stale seed/lesson data — lesson JSONB missing keys, empty furigana tokens, orphaned vocab UUIDs, unparseable conjugation paths
  - Cross-song session leakage — isSessionForSong guard verified (Song A session never appears on Song B exercise tab)
  - Premium gate bypass — checkExerciseAccess() is the single gate; free users can't reach premium exercises via URL/state manipulation
  - Geo-restricted / missing YouTube videos fail gracefully (no infinite spinner, clear user message)

### Test layering
- **All four layers in scope:**
  - Playwright E2E (browser) — real app, highest-confidence path
  - Vitest unit (pure TS) — exercise generator, deriveStars, checkExerciseAccess, distractor picker, etc.
  - Node integration (DB + API) — API route + Drizzle query tests against Postgres, no browser
  - Seed/content QA — extend existing 06-qa-agent.ts: lesson coverage, furigana completeness, UUID integrity, geo-check across catalog

### YouTube embed handling
- **Real YouTube iframe in E2E tests** — accept network/geo flakiness cost for highest confidence that sync logic actually works against the live embed. No mocked window.YT / postMessage stubs for the critical sync path.

### Run environment + cadence
- Speed budget: **under 15 minutes for the full suite** — comfortable ceiling that enables deep E2E across many flows; suite is intentional, not run on every save
- Failure reporting: **terminal-first summary** — concise failing-test list in terminal with file paths. Playwright's HTML report is opt-in, not default. Fits CLI-heavy workflow and CI logs equally
- **Zero-tolerance flaky-test policy** — any intermittent failure quarantines the test until root-caused and fixed. No auto-retry hiding real bugs. Suite trustworthiness > velocity

### Claude's Discretion
- Bug routing across layers (strict separation vs redundant E2E safety net vs tiered smoke/deep) — Claude picks based on what's idiomatic for Next.js + Drizzle + Playwright
- Existing test infra starting point — extend vs consolidate — Claude inspects current vitest.config, existing 08-02 generator tests, and 06-qa-agent.ts and decides
- Playwright Page Object Model vs inline selectors — Claude picks based on final suite size (likely POM once >20 tests)
- Test database strategy (dedicated test DB vs dev DB with test users vs ephemeral per-run) — Claude picks what's cheapest to maintain given the current seed pipeline
- Auth state handling (storage-state fixture vs per-test login vs test-user factory) — Claude picks based on the existing auth stack
- FSRS / progress state reset between tests (truncate vs isolated user per test vs transactional rollback)
- Run environment (local-only vs GitHub Actions on PR vs nightly cron) — Claude picks based on current CI state and solo-dev pace

</decisions>

<specifics>
## Specific Ideas

- Terminal-first reporting aligns with the CLI-heavy workflow already in use for seed scripts (scripts/seed/*.ts, npm run audit:geo:replace, etc.) — QA output should look like those scripts, not a web dashboard
- Zero-flake policy is a strong stance: the user values a trustworthy small suite over a large suite with retries masking real bugs
- Real YouTube iframe in E2E signals the sync logic (verse highlight within 250ms of playback) is considered a core value prop that must not be mocked away
- Catalog state during test runs: current dev DB has 116 song versions post-Phase-8-04 plus 60 pending-whisper TV rows — tests must be deterministic about which subset they run against

</specifics>

<deferred>
## Deferred Ideas

- Visual regression / screenshot diffing across the lesson panel — not raised, could land in a later phase if UI churn becomes a problem
- Load / performance testing (many concurrent users, long-session memory leaks) — out of scope for 08.1
- Accessibility / screen-reader testing — out of scope; would be its own phase if prioritized
- TV-pack exercise coverage — currently 60 TV rows have lesson=NULL pending WhisperX batch; the QA suite should not block on TV coverage and should skip gracefully when a song has no lesson

</deferred>
