<task type="auto">
  <name>Task 2: Flake quarantine convention + docs</name>
  <files>README-testing.md, playwright.config.ts, vitest.config.ts</files>
  <action>
    Define the quarantine convention:
    - Tests flagged as flaky get a `@kb-quarantine` string in their test title, e.g. `test("[kb-quarantine] resume session after long idle", …)`
    - Default runs FILTER OUT quarantined tests:
      - In playwright.config.ts, add `grepInvert: /\[kb-quarantine\]/` so quarantined tests are skipped by default
      - In vitest.config.ts, add `test.exclude` or a `test.name` filter via a setup file; simplest: add a `describe.skipIf(!process.env.KB_RUN_QUARANTINE)` helper exported from tests/support/quarantine.ts
    - A `test:quarantine` npm script runs ONLY quarantined tests (for debugging the ones that are flaky): `playwright test --grep '\\[kb-quarantine\\]'`. Add it to package.json.
    - Rule (enforced by convention, not code): quarantine tag is a temporary measure. Every quarantined test MUST have a TODO comment with a date + reason. If more than 3 tests are ever quarantined at once, the suite is considered degraded and the team must drain the queue before shipping new features. Document this in README-testing.md.

    Create README-testing.md (keep it under 100 lines):
    - Section: "Running the suite"
      - Prerequisites: TEST_DATABASE_URL set
      - **Ordering requirement:** `test:all` prepends `test:seed` automatically (set in plan 01's package.json). If you run individual layers manually (e.g. `test:integration`), run `npm run test:seed` first — the suite assumes the 3 seeded slugs exist and the test user's progress tables are empty. Running without a fresh seed is the single most common cause of spurious failures.
      - Commands: test:seed (run first), test:unit, test:integration, test:e2e, test:qa, test:all, test:measure, test:quarantine
    - Section: "Test DB"
      - One-sentence explanation: dedicated Neon/Postgres DB, TEST_USER_ID constant, reset helpers
    - Section: "Zero-flake policy"
      - retries:0 is non-negotiable
      - Flaky tests go to @kb-quarantine and must be fixed within 1 sprint
      - Max 3 quarantined at a time
    - Section: "Speed budget"
      - Full suite < 15 minutes
      - PR runs skip E2E; nightly runs everything
    - Section: "Layers" — one-liner per layer (unit, integration, e2e, seed-qa)
    - Section: "Test hooks" — list of data-attributes added to production code (from plans 05/06/07 SUMMARYs) so future contributors know what's instrumented for tests

    Edit playwright.config.ts + vitest.config.ts to apply grepInvert / skipIf. Keep changes minimal.
  </action>
  <verify>
    `npm run test:e2e` runs with no `[kb-quarantine]` tests. Manually add one with the tag to a scratch spec, confirm it's skipped by default, then `npm run test:quarantine` picks it up, then delete the scratch spec.
    README-testing.md exists and covers all 5 sections in under 100 lines.
  </verify>
  <done>

Installs Playwright browsers with `npx playwright install chromium --with-deps`. Uploads test-results/ + playwright-report/ as artifacts on failure.
    - Both jobs need TEST_DATABASE_URL as a repo secret (document in README-testing.md that the user must add it in GitHub Settings → Secrets). Also DATABASE_URL if required by any action imports; seed-test-db.ts requires only TEST_DATABASE_URL.
    - Fail-fast behavior: `continue-on-error: false` default; zero retries.
    - Include a step to run `npm run test:seed` before tests so the test DB catalog is fresh.
    - Keep YAML minimal — don't pre-optimize for matrix builds or multiple Node versions.

    If the repo already has another GH Actions workflow (check with a quick ls of .github/workflows/), make this one additive — don't overwrite.
