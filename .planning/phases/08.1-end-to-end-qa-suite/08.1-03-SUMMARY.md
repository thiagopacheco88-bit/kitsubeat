## Decisions Made

- **Single global setupFiles entry** (not vitest projects per-glob). Simpler than splitting unit vs integration into separate vitest projects; the side effect (reading .env, redirecting DATABASE_URL) is benign for unit tests in `src/` since they don't touch the DB.
- **describe.skip guard above the helper throw.** `tests/support/test-db.ts::getTestDb()` throws when `TEST_DATABASE_URL` is unset (correct behavior for the helper). At the test-file level we add `const describeIfTestDb = HAS_TEST_DB ? describe : describe.skip` so the runner reports a clean skip rather than an import-time error. Both layers cooperate: the helper protects against accidental dev-DB writes, the per-file guard keeps the runner green when the operator hasn't provisioned the test DB yet.
- **Unknown-exercise-type contract** is asserted via TWO subcases inside one `it()` — mixed (one unknown + one valid → only valid counted) and all-unknown (no group accuracy populated, but row still inserted with sessions_completed=1, completion_pct=15, stars=0). The plan called for both; combining into one test keeps the suite tidy and the resetTestProgress mid-test makes the second subcase truly independent.
- **Direct route handler invocation** (`import { GET } from "@/app/..."` + `new NextRequest(new URL(...))`). Cuts ~3s of Next.js dev-server startup per run and exercises the same handler the framework would invoke. saveSessionResults is a server action — direct import works the same way.
- **Drizzle `.execute(sql)` result shape handling.** The neon-http driver returns either an array directly OR an object `{ rows: [] }` depending on the query. All test files use a defensive `const rows = Array.isArray(raw) ? raw : (raw.rows ?? []);` pattern. Documented as a tech-stack pattern.
- **Batch test reuses songVersionId** if the test DB only has one seeded `song_versions` row. The Map-keying assertion still holds with one entry; the test does not regress when the catalog is minimally seeded.
- **20-answer batches in queries-progress.test.ts** to land exact 0.85 / 0.70 accuracies via integer correct counts (17/20, 14/20). Avoids floating-point comparison noise around the deriveStars `>= 0.80` threshold.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Added `setupFiles` to global vitest config (not integration-only project)**
- **Found during:** Task 1 design
- **Issue:** The plan suggested either `vi.hoisted`, top-level `beforeAll`, or a `setupFiles` scoped to integration via project config merge. vitest 4 supports projects, but configuring per-glob setupFiles requires splitting into a vitest project file — overkill for one setup line. Inline `vi.hoisted` would scatter the env-redirect logic across 4 files.
- **Fix:** Added a single `setupFiles: ["./tests/integration/setup.ts"]` entry in vitest.config.ts. Documented inline why it's safe for unit tests (they don't touch DATABASE_URL). One source of truth instead of 4.
- **Files modified:** `vitest.config.ts`, `tests/integration/setup.ts`
- **Verification:** `npm run test:integration` runs setup.ts; `npm run test:unit` is unaffected (setup runs but the env redirect is silently a no-op when TEST_DATABASE_URL is unset, plus unit tests don't import the db module).
- **Committed in:** `ae96508`

**2. [Rule 2 - Missing Critical] Added describe.skip guard to all 4 test files**
- **Found during:** Task 1 implementation (env env-constraint reading)
- **Issue:** The plan's tests would fail at import time when `TEST_DATABASE_URL` is unset because `getTestDb()` throws as soon as a test calls it (or the route handler does). The env-constraint section explicitly required either `describe.skip` or relying on the helper's throw; we picked the cleaner option since "throws on import" produces an ugly stack trace in vitest output.
- **Fix:** Each test file declares `const HAS_TEST_DB = !!process.env.TEST_DATABASE_URL;` and uses `describeIfTestDb` (= `describe` or `describe.skip`) for its outer block. Result: skip is reported as a normal skip line.
- **Files modified:** All 4 integration test files.
- **Verification:** `npm run test:integration` reports `Tests 16 skipped (16)` cleanly with no errors.
- **Committed in:** `ae96508`, `67a1853`, `7cc0d81`

**3. [Rule 2 - Missing Critical] Defensive `Array.isArray(raw) ? raw : (raw.rows ?? [])` pattern for drizzle execute() results**
- **Found during:** Task 1 (jlpt-pool route returns `rows.rows ?? rows`, hint that the shape varies)
- **Issue:** The neon-http driver's `db.execute(sql)` return shape varies — some queries return an array directly, others return `{ rows: [...] }`. Without the defensive unwrap, the parity check in admin-songs-api.test.ts and the seeded-id lookup in save-session-results.test.ts would silently get `undefined`, masking failures.
- **Fix:** Helper inline pattern in every place that calls `db.execute(sql)`. Documented as a project pattern in tech-stack.patterns.
- **Files modified:** all 4 test files (where applicable).
- **Committed in:** `ae96508`, `67a1853`, `7cc0d81`
