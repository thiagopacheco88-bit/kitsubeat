    **Investigation result (locked by planner, not executor):** `src/app/actions/exercises.ts::saveSessionResults` currently writes ONLY `user_song_progress`. The `user_vocab_mastery` table exists in `src/lib/db/schema.ts` with FSRS columns (`stability`, `difficulty`, `state`, `due`, `reps`, `lapses`, `intensity_preset`) but NO server action writes to it. `src/lib/fsrs-presets.ts` exists but is unused by saveSessionResults.

    CONTEXT (08.1) explicitly locks "Progress + FSRS update (user_song_progress + user_vocab_mastery rows write/update)" as MUST-test coverage. We therefore MUST wire the write here — a `.fixme` on a locked-coverage bullet is not acceptable. Wiring is a narrow connection of existing infrastructure (schema + fsrs-presets + per-answer vocab_item_id already carried on AnswerRecord) to an existing server action, NOT a new feature.

    **Step 1 — Extend saveSessionResults (src/app/actions/exercises.ts):**
    - Extend `AnswerRecord` to include `vocabItemId: string` (already carried on the client as `Question.vocabItemId` — thread it through).
    - After the existing user_song_progress upsert block, loop over unique `vocabItemId` values in `answers`.
    - For each `(userId, vocabItemId)` pair, upsert a row in `userVocabMastery`:
      - On insert: use `fsrs-presets.ts` defaults (intensity_preset = 'default' or whatever the preset module exposes) for `stability`, `difficulty`, `state=0 (new)`, `due=NOW()`, `reps=1`, `lapses=0` (or 1 if the answer was wrong).
      - On conflict (uses the existing `user_vocab_mastery_user_vocab_unique` constraint on `(user_id, vocab_item_id)`): increment `reps`, increment `lapses` if the answer was wrong, update `due` / `stability` / `difficulty` using a simple FSRS-lite update (if a library like `ts-fsrs` is in package.json — check — use it; otherwise use the minimal heuristic encoded in `fsrs-presets.ts`: correct → double stability, next due = NOW() + stability days; wrong → halve stability, reset due to NOW() + 10 min).
    - If `ts-fsrs` or equivalent is NOT installed, do NOT add a new dependency as part of this task. Use the stability-doubling heuristic above and mark a clear comment `// TODO Phase 9: swap for proper FSRS algorithm`. This is acceptable because the test assertions below (fields non-null, row-per-vocab) are satisfied by any algorithm.
    - Preserve the existing return shape of `saveSessionResults`.

    **Step 2 — Create tests/e2e/exercise-progress-fsrs.spec.ts:**
    - Uses testUser fixture + imports getTestDb for direct DB SELECTs after the UI flow
    - `test("completion_pct increments after session")`:
      1. Reset DB
      2. Query initial count of user_song_progress for TEST_USER_ID → expect 0
      3. Run full short session (correct answers)
      4. Query again → expect 1 row with `completion_pct = 15`, `sessions_completed = 1`
      5. Run another short session → `completion_pct = 30`, `sessions_completed = 2`

    - `test("user_vocab_mastery rows written for answered vocab")` — **HARD assertion, no `.fixme`**:
      1. Reset (resetTestProgress must include `DELETE FROM user_vocab_mastery WHERE user_id = $1` — add that to tests/support/test-db.ts as part of this task if not already included; it's a trivial extension of plan 01's reset helper).
      2. Run short session answering correctly
      3. Query `SELECT count(*) FROM user_vocab_mastery WHERE user_id = 'test-user-e2e'` → expect ≥ 5 rows (one per unique vocab_item_id answered — a 10-question short session covers 5+ distinct vocab entries).
      4. Each row has FSRS fields populated: `stability` non-null, `difficulty` non-null, `state` non-null (integer), `due` non-null (timestamp).
      5. Assert uniqueness: `SELECT COUNT(*) FROM user_vocab_mastery GROUP BY user_id, vocab_item_id HAVING COUNT(*) > 1` returns zero rows (the unique constraint holds).

    - `test("repeat session updates (not duplicates) user_vocab_mastery")`:
      1. Reset + run session once → N rows
      2. Run another session on the same song (overlapping vocab) → still N rows (or N+delta where delta = new-vocab-count), `reps` incremented on existing rows.

    - `test("SongCard on /songs reflects completion_pct")`:
      1. Reset, run a session
      2. Navigate to /songs
      3. Find the SongCard for again-yui (by href)
      4. Assert completion display (inspect SongCard for the % element) shows the new percentage
