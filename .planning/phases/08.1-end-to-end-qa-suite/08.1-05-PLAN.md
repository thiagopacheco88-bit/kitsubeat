<objective>
Playwright E2E tests for the v1.0 player experience. Uses the REAL YouTube iframe (per CONTEXT locked decision) — no postMessage stubs for the critical sync path. Covers song load, lesson toggles, token popup, verse timing sync with seek/skip, and the grammar + vocabulary panels.

Purpose: The player is half of the product. Integration + unit tests cannot catch layout regressions, hydration issues, or YouTube Player API race conditions. This plan locks those down.

Output:
- 4 E2E spec files, each with a narrow focus, running against a fixed seeded song (`again-yui`) whose data we know well from the existing tests/app.spec.ts baseline

**Note on verse-highlight timing tolerance:** CONTEXT describes a perception-spec target of ~250ms for verse-highlight sync. This plan's assertion uses a 1500ms ceiling deliberately — the tight 250ms figure is the aspirational UX target measured in isolation, whereas the E2E test runs against a real YouTube iframe with real network + paint jitter and must remain stable on CI hardware. 1500ms is the enforced product regression floor (if we slip below it, the bug is obvious to users); 250ms is a manual perception check we do NOT automate to avoid false flakes. This relaxation is intentional and should NOT be tightened without a separate performance plan.
</objective>
