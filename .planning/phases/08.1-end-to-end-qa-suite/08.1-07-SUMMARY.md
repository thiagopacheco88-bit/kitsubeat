## YouTubeEmbed graceful failure UI — exact source

**Embed states** (single source of truth):
```ts
type EmbedState = "loading" | "ready" | "error";
const WATCHDOG_MS = 15_000;
```

**State transitions:**
- `onReady` (YT.Player event) → `setEmbedState("ready")`
- `onError` (YT.Player event, any code) → `setEmbedState("error")`
- 15s watchdog → `setEmbedState((prev) => prev === "loading" ? "error" : prev)` (functional form prevents clobbering a late-firing onReady)
- `currentId` / `version` change → reset to `"loading"` and re-arm the watchdog

**Fallback UI** (locked copy):
- Heading: `"Video unavailable"`
- Body: `"This video may be geo-restricted or removed. The lyrics and lesson content below still work."`
- Container: `data-yt-state="error"` + Tailwind `aspect-video` so layout doesn't shift

## Decisions Made

(See `key-decisions` in frontmatter for the full set.)

Highlights:

- **Route-intercept for geo-fallback (not a real blocked video).** Deterministic, doesn't depend on YouTube's regional availability staying constant, and exercises both watchdog (abort path) and onError handler (404 path) — the two failure modes a real geo-block can produce.
- **15s watchdog with functional-state-update guard.** `setEmbedState((prev) => prev === "loading" ? "error" : prev)` prevents the watchdog from racing with a late onReady — if onReady fires first, prev is already "ready" and the watchdog is a no-op.
- **Locked fallback copy committed in BOTH the production component AND the spec.** A future copy change must update both files in the same PR. Brittle by design — a fuzzy match would let "Video unavailable" silently become "Error 7401" without anyone noticing.
- **Single-gate invariant colocated in regression-stale-lesson-data.test.ts.** The plan declared one integration file; an extra `regression-single-gate.test.ts` would have shipped an undeclared artifact and failed the plan-08.1-08 manifest audit.
- **Premium-gate `.fixme` is intentional, not a test bug.** The integration test save-session-results.test.ts ("unknown exercise types are silently dropped (no throw, not counted)") already locks today's behavior; the .fixme serves as the living spec for the Phase 10 follow-up that adds checkExerciseAccess routing inside the server action.
- **Cross-song leak round-trip (BOTH directions).** Asserting only "Song A -> Song B refuses" would prove rejection, but not preservation. Round-trip proves the persisted state survived intact so the user can switch songs without losing progress.
