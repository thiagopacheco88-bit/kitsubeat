# Phase 08.3 Plan 01: Schema Migration + Type Foundation Summary

**One-liner:** Nullable jsonb mnemonic + kanji_breakdown columns on vocabulary_items, with KanjiCharEntry/KanjiBreakdown types and carry-through in Question.

## What Was Built

### SQL Applied
`drizzle/0004_vocab_enrichment.sql` adds two nullable jsonb columns to `vocabulary_items`:
- `"mnemonic" jsonb` — per-language Localizable object (or null if not yet enriched)
- `"kanji_breakdown" jsonb` — KanjiBreakdown-shaped object with characters array (or null for kana-only words)

Both use `ADD COLUMN IF NOT EXISTS` making the migration idempotent. Verified applied to the live Neon DB; re-run exits 0 (columns already exist, PostgreSQL no-ops silently).

### Drizzle Schema (src/lib/db/schema.ts)
Two nullable jsonb columns added to the `vocabularyItems` pgTable, right after `meaning`. Drizzle infers both as `unknown` at the type boundary (expected behavior for jsonb — callers narrow via Zod at use sites in Wave 2+).

### New Types (src/lib/types/lesson.ts)
Inserted between the `localize()` block and `Token` interface:
- `KanjiCharEntry` — single kanji character with `meaning: Localizable`, `on_yomi`, `kun_yomi`, `jlpt_level: JlptLevel | null`, `radical_hint: Localizable`
- `KanjiBreakdown` — array of `KanjiCharEntry` + optional `compound_note: Localizable`

`VocabEntry` extended with two optional fields after `vocab_item_id`:
- `mnemonic?: Localizable`
- `kanji_breakdown?: KanjiBreakdown | null`

### Generator carry-through (src/lib/exercises/generator.ts)
- Import line extended with `KanjiBreakdown` and `Localizable`
- `Question` interface extended with `mnemonic?: Localizable` and `kanji_breakdown?: KanjiBreakdown | null` (placed before `verseRef?:`)
- `makeQuestion()` return object now includes `mnemonic: vocab.mnemonic` and `kanji_breakdown: vocab.kanji_breakdown ?? null`
