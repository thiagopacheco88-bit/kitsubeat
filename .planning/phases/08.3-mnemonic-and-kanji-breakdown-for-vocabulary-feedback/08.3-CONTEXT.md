# Phase 08.3: Mnemonic and Kanji Breakdown for Vocabulary Feedback - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Enrich the vocabulary identity layer with two learning aids — a per-word mnemonic and a per-word kanji breakdown — generate them via inline LLM calls into existing data, and surface them inside the exercise FeedbackPanel "More" accordion. Scope covers: schema additions, content generation script, backfill across all existing songs, and FeedbackPanel rendering. Does **not** cover cross-song kanji aggregation, kanji-level mastery tracking, or new exercise types (those belong in later phases).

</domain>

<decisions>
## Implementation Decisions

### Storage location
- Mnemonic and kanji_breakdown live on the `vocabulary_items` table (Phase 7), **not** inside the per-song lesson JSONB.
- Word-level data is generated and stored once per UUID — same word in 5 songs reuses the same mnemonic.
- FeedbackPanel reads via the existing `vocab_item_id` UUID linkage on `VocabEntry`.
- Approximate scale: ~705 vocabulary_items rows to enrich (not ~1200 per-song entries).

### Mnemonic style
- One short sentence, ~10–15 words. Quick scan, fits inline in the panel.
- Tone: playful + visual — wordplay, vivid imagery, sound-alikes. Not anime-anchored, not etymological.
- Stored as `Localizable` (EN/PT/ES) — matches existing `meaning` field shape; consumed via `localize(mnemonic, translationLang)`.
- Shown only for the question's target vocab. Distractor vocab in the wrong-pick callout does **not** render its own mnemonic (keeps panel focused).

### Kanji breakdown shape
- Unit: per character + brief compound note for multi-kanji words (e.g., 鉄道 = 鉄 "iron" + 道 "path" → "iron road = railway").
- Per-character fields: meaning, on'yomi, kun'yomi, radical/component hint.
- Per-character JLPT level included.
- Meaning + compound note + radical hint stored as `Localizable` (EN/PT/ES). Readings (on/kun) are kana — locale-agnostic, single string.
- No cross-references to other songs/words (that's Phase 11 territory).

### Display behavior (FeedbackPanel)
- Both mnemonic and kanji_breakdown render **inside the existing "More" accordion**, alongside `detailedExplanation`.
- "More" accordion **persists open across questions within a session** (currently resets per question). Implies a session-store flag rather than local component state.
- For kana-only words (no kanji in surface), the kanji_breakdown section is hidden entirely. Mnemonic still renders.
- Visual layout for the kanji breakdown grid/list: **Claude's discretion** — pick what matches the rest of the panel.

### Generation strategy
- Inline LLM generation in conversation (per user's standing preference — **not** Claude Batch API).
- Two-track approach:
  1. New targeted enrichment script (e.g., `scripts/seed/11-enrich-vocab.ts`) that reads `vocabulary_items` rows missing the new fields and fills only those.
  2. Update prompt in `scripts/seed/03-generate-content.ts` so future songs include the fields from the start.
- Existing lesson content is **not** regenerated — surgical enrichment only.
- Failure handling: per-item commit + resume. Each `vocabulary_items` row is updated immediately after generation; re-runs skip rows that already have a populated mnemonic. Slowest but fully idempotent.

### QA gate
- Coverage check: every kanji-bearing row has both fields populated; every kana-only row has mnemonic populated.
- Automated quality heuristics:
  - Mnemonic word count within configured bounds (e.g., 5–25 words).
  - `kanji_breakdown` character count matches kanji count in `surface`.
  - No empty `meaning` fields inside per-character entries.
- Informal: eyeball ~10 random rows before sign-off (not the gate, just a sanity pass).

### Claude's Discretion
- Visual layout of the kanji breakdown (stacked rows vs card grid vs compact inline) — pick what matches existing FeedbackPanel rhythm.
- Exact word-count bounds for mnemonic QA heuristic.
- Concrete prompt wording for the enrichment script — must satisfy decisions above (length, tone, fields, locale shape).
- Whether `radical/component hint` is one short string or its own sub-object — pick what serializes cleanly.
- Migration mechanics: column types (jsonb vs text + jsonb), nullability, index needs.

</decisions>

<specifics>
## Specific Ideas

- Consume via existing `localize()` helper in `src/lib/types/lesson.ts` so EN/PT/ES switching "just works" with the current `translationLang` from `PlayerContext`.
- Kanji breakdown should bridge per-character meaning to whole-word meaning (compound note exists for that reason — not just a list of kanji glosses).
- Persistence behavior of the "More" accordion is the one ergonomic ask: opening it on Q3 should leave it open on Q4 in the same exercise session.

</specifics>

<deferred>
## Deferred Ideas

- Cross-song kanji aggregation ("this kanji also appears in: …") — Phase 11 (Cross-Song Vocabulary).
- Per-kanji mastery tracking — would need its own FSRS schema, out of scope.
- Anime-anchored mnemonics ("think how Naruto says X") — considered and rejected for tone; could be revisited later as a separate flavor pass.
- Kana-origin notes for kana-only words (onomatopoeia/mimetic flagging) — not adopted; kanji block hidden cleanly instead.
- Showing the distractor's mnemonic in the wrong-pick callout — rejected to keep panel focused.

</deferred>
