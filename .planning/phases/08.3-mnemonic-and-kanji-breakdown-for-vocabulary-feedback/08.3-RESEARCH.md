## User Constraints (from CONTEXT.md)

### Locked Decisions

**Storage location**
- Mnemonic and kanji_breakdown live on the `vocabulary_items` table (Phase 7), not inside the per-song lesson JSONB.
- Word-level data is generated and stored once per UUID — same word in 5 songs reuses the same mnemonic.
- FeedbackPanel reads via the existing `vocab_item_id` UUID linkage on `VocabEntry`.
- Approximate scale: ~705 vocabulary_items rows to enrich.

**Mnemonic style**
- One short sentence, ~10–15 words. Quick scan, fits inline in the panel.
- Tone: playful + visual — wordplay, vivid imagery, sound-alikes. Not anime-anchored, not etymological.
- Stored as `Localizable` (EN/PT/ES) — matches existing `meaning` field shape; consumed via `localize(mnemonic, translationLang)`.
- Shown only for the question's target vocab. Distractor vocab in wrong-pick callout does NOT render its own mnemonic.

**Kanji breakdown shape**
- Unit: per character + brief compound note for multi-kanji words (e.g., 鉄道 = 鉄 "iron" + 道 "path" → "iron road = railway").
- Per-character fields: meaning, on'yomi, kun'yomi, radical/component hint.
- Per-character JLPT level included.
- Meaning + compound note + radical hint stored as `Localizable` (EN/PT/ES). Readings (on/kun) are kana — locale-agnostic, single string.
- No cross-references to other songs/words.

**Display behavior (FeedbackPanel)**
- Both mnemonic and kanji_breakdown render inside the existing "More" accordion, alongside `detailedExplanation`.
- "More" accordion persists open across questions within a session (currently resets per question). Implies a session-store flag rather than local component state.
- For kana-only words (no kanji in surface), the kanji_breakdown section is hidden entirely. Mnemonic still renders.

**Generation strategy**
- Inline LLM generation in conversation (NOT Claude Batch API).
- Two-track approach:
  1. New targeted enrichment script (`scripts/seed/11-enrich-vocab.ts`) that reads `vocabulary_items` rows missing the new fields and fills only those.
  2. Update prompt in `scripts/seed/03-generate-content.ts` so future songs include the fields from the start.
- Existing lesson content is NOT regenerated — surgical enrichment only.
- Failure handling: per-item commit + resume. Each `vocabulary_items` row is updated immediately after generation; re-runs skip rows that already have a populated mnemonic. Fully idempotent.

**QA gate**
- Coverage check: every kanji-bearing row has both fields populated; every kana-only row has mnemonic populated.
- Automated quality heuristics:
  - Mnemonic word count within configured bounds.
  - `kanji_breakdown` character count matches kanji count in `surface`.
  - No empty `meaning` fields inside per-character entries.
- Informal: eyeball ~10 random rows before sign-off (not the gate, just sanity).

### Claude's Discretion

- Visual layout of the kanji breakdown (stacked rows vs card grid vs compact inline) — pick what matches existing FeedbackPanel rhythm.
- Exact word-count bounds for mnemonic QA heuristic.
- Concrete prompt wording for the enrichment script — must satisfy decisions above.
- Whether `radical/component hint` is one short string or its own sub-object — pick what serializes cleanly.
- Migration mechanics: column types (jsonb vs text + jsonb), nullability, index needs.

### Deferred Ideas (OUT OF SCOPE)

- Cross-song kanji aggregation ("this kanji also appears in: …") — Phase 11.
- Per-kanji mastery tracking.
- Anime-anchored mnemonics.
- Kana-origin notes for kana-only words (onomatopoeia/mimetic flagging).
- Showing the distractor's mnemonic in the wrong-pick callout.

## Summary

Phase 08.3 extends the `vocabulary_items` table with two new JSONB columns (`mnemonic` and `kanji_breakdown`), generates content for ~705 existing rows via a new enrichment script using inline Claude API calls, and surfaces the content inside the FeedbackPanel "More" accordion.

All three layers — database, generation, and UI — are well-understood from direct code inspection. The migration path is manual SQL (not drizzle-kit generate), following the established pattern in `drizzle/`. The enrichment script follows the exact commit-per-row pattern from `backfill-vocab-identity.ts`. The FeedbackPanel has a clean insertion point for the new accordion content; the only structural change is lifting the `showMore` boolean into the Zustand `exerciseSession` store for cross-question persistence.

No new npm dependencies are required. The Anthropic SDK (`@anthropic-ai/sdk@^0.82.0`) and inline Messages API call pattern already exist in `scripts/seed/03b-generate-content-sequential.ts`. There is no kanji reference library in the codebase — all kanji data (on'yomi, kun'yomi, radical hints) must be LLM-generated, which is consistent with the existing approach.

**Primary recommendation:** Write the migration file first, apply it, then build the enrichment script using the `03b` inline call pattern, then update FeedbackPanel with accordion lift.

### Pattern 1: Manual SQL Migration File

**What:** Add two nullable JSONB columns to `vocabulary_items` with a single migration file.

**When to use:** Any time the schema changes — drizzle-kit generate is NOT used in this project due to interactive mode issues.

**Established pattern from `drizzle/0002_data_foundation.sql`:**

```sql
-- drizzle/0004_vocab_enrichment.sql
-- Adds mnemonic and kanji_breakdown to vocabulary_items (Phase 08.3)

ALTER TABLE "vocabulary_items"
  ADD COLUMN IF NOT EXISTS "mnemonic" jsonb;
--> statement-breakpoint
ALTER TABLE "vocabulary_items"
  ADD COLUMN IF NOT EXISTS "kanji_breakdown" jsonb;
```

Applied via: `npx tsx scripts/apply-migrations.ts 0004_vocab_enrichment.sql`

The `apply-migrations.ts` script splits on `--> statement-breakpoint`, handles `already exists` gracefully, and exits with code 1 on failure.

**Both columns are nullable.** A null `mnemonic` means "not yet enriched". The enrichment script skips rows where `mnemonic IS NOT NULL`.

**No index needed** on these columns — enrichment queries filter on `mnemonic IS NULL` which works as a sequential scan on 705 rows.

### Pattern 7: Fetching Enrichment Data to FeedbackPanel

The enrichment data lives on `vocabulary_items`, not in the lesson JSONB. The `Question` object (from `generator.ts`) carries `vocabItemId` (UUID). The FeedbackPanel gets `question` as a prop.

**Fetch path options:**
1. **API route** — `GET /api/vocab/[id]/enrichment` → returns `{ mnemonic, kanji_breakdown }` — clean separation, but adds a per-question network call.
2. **Pre-loaded with the lesson** — the server-side lesson-loading code (in `src/app/songs/[slug]/page.tsx`) joins `vocabulary_items` to get enrichment fields, attaches them to `VocabEntry` in the lesson payload.
3. **Embed in `VocabInfo` type** — extend `VocabInfo` in `generator.ts` to carry optional `mnemonic` and `kanji_breakdown`, populate from the lesson data that already includes them.

**Recommended: Option 2 — pre-load with lesson.** The lesson is already fetched server-side at page render. Joining `vocabulary_items` to add `mnemonic` and `kanji_breakdown` onto each `VocabEntry` costs one additional SELECT at page load, not per question. The data then flows into `buildQuestions()` → `Question.vocabInfo` without any async calls in the feedback UI.

**Concretely:** Extend `VocabEntry` in `src/lib/types/lesson.ts`:
```typescript
interface VocabEntry {

  /** Enrichment fields — populated at page load from vocabulary_items join */

  kanji_breakdown?: KanjiBreakdown | null;

```

Extend `VocabInfo` in `generator.ts`:

  kanji_breakdown?: KanjiBreakdown | null;

The page-level data fetch joins these fields from `vocabulary_items` by UUID and injects them. No API route needed; no per-question fetch.

## Common Pitfalls

### Pitfall 1: `radical_hint` as sub-object vs string

**What goes wrong:** Defining `radical_hint` as `{ name: string; component: string }` forces the LLM to fill two fields for one concept and makes the UI render more complex.

**Why it happens:** Over-engineering the type.

**How to avoid:** Keep it a single short string: `"fire + person"` or `"hand radical + journey"`. The UI just renders it inline. Claude's discretion is to use a string (locked in recommended schema above).

**Warning signs:** If the Zod schema for `radical_hint` has nested keys, simplify it.

### Pitfall 2: `compound_note` for single-kanji words

**What goes wrong:** Including `compound_note` in the LLM prompt for single-kanji words causes hallucinated notes like "this word means X alone".

**How to avoid:** In the enrichment prompt, explicitly instruct: "Only include `compound_note` if the word contains 2+ kanji characters." In the Zod schema, `compound_note` is `.optional()` — undefined is valid for single-kanji words.

### Pitfall 3: Unicode iteration of kanji characters

**What goes wrong:** `surface.split("")` splits some emoji or extended Unicode into surrogate pairs, giving wrong character count.

**How to avoid:** Always use `[...surface]` (spread into array) to correctly handle Unicode code points. The kanji count for QA validation must use `[...surface].filter(c => KANJI_RE.test(c)).length`.

### Pitfall 4: `moreAccordionOpen` persisting across songs

**What goes wrong:** If `moreAccordionOpen` is in the Zustand store and persists to localStorage, it carries over when the user navigates to a different song.

**How to avoid:** Reset `moreAccordionOpen` to `false` inside `startSession()`. The flag should survive within a session (cross-question) but reset between sessions.

```typescript
startSession: (songVersionId, questions, mode) =>
  set({
    songVersionId, questions, currentIndex: 0, answers: {}, startedAt: Date.now(), mode,
    moreAccordionOpen: false,  // reset on new session
  }),
```

### Pitfall 5: Enrichment script re-running on already-enriched rows

**What goes wrong:** If the skip condition is `mnemonic IS NULL` but the column stores a serialized `null` JSON value (not SQL NULL), rows get re-enriched every run.

**How to avoid:** Ensure the UPDATE always writes a real JSONB value (never JSON `null` for mnemonic). For kana-only words: `mnemonic = <generated>`, `kanji_breakdown = null` (SQL NULL for the kanji_breakdown column is fine since it has no kanji; mnemonic is always populated). Skip condition: `WHERE mnemonic IS NULL` — SQL NULL, not `'null'::jsonb`.

### Pitfall 6: FeedbackPanel rendering kanji_breakdown for kana-only words

**What goes wrong:** Kana-only words like サクラ have no kanji, so kanji_breakdown will be NULL. Rendering it anyway produces an empty section.

**How to avoid:** Guard at render time:
```tsx
{vocab.kanji_breakdown && vocab.kanji_breakdown.characters.length > 0 && (
  <KanjiBreakdownSection breakdown={vocab.kanji_breakdown} lang={translationLang} />
)}
```

This is safe even if the DB has `kanji_breakdown = null` — null/undefined both evaluate to falsy.

### Pitfall 7: "More" accordion condition change breaking existing behavior

**What goes wrong:** Currently the accordion only renders when `question.detailedExplanation` is truthy. After this phase, it also renders for mnemonic/kanji_breakdown. If a question has none of these, the accordion button should not appear.

**How to avoid:** Update the render condition to:
```tsx
const hasMoreContent = question.detailedExplanation || vocab.mnemonic || vocab.kanji_breakdown;
if (!hasMoreContent) return null; // Don't render accordion at all
```

## Code Examples

### Enrichment Script Skeleton

```typescript
// scripts/seed/11-enrich-vocab.ts
import { config } from "dotenv";
config({ path: ".env.local" });

import Anthropic from "@anthropic-ai/sdk";
import pLimit from "p-limit";
import { eq, isNull } from "drizzle-orm";
import { getDb } from "../../src/lib/db/index.js";
import { vocabularyItems } from "../../src/lib/db/schema.js";
import { VocabEnrichmentSchema, ENRICH_JSON_SCHEMA } from "../types/enrich.js";

const MODEL = "claude-haiku-4-5";
const CONCURRENCY = 5;
const KANJI_RE = /[\u4e00-\u9fff]/;

async function main() {
  const db = getDb();
  const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY!, maxRetries: 3 });
  const limit = pLimit(CONCURRENCY);

  // Load rows missing mnemonic
  const rows = await db
    .select()
    .from(vocabularyItems)
    .where(isNull(vocabularyItems.mnemonic));

  console.log(`Enriching ${rows.length} vocabulary items...`);

  let succeeded = 0; let failed = 0;
  const tasks = rows.map((row) => limit(async () => {
    try {
      const hasKanji = KANJI_RE.test(row.dictionary_form);
      const prompt = buildEnrichmentPrompt(row, hasKanji);
      const response = await client.messages.create({
        model: MODEL,
        max_tokens: 1024,
        messages: [{ role: "user", content: prompt }],
        // eslint-disable-next-line @typescript-eslint/no-explicit-any
        output_config: { format: { type: "json_schema", json_schema: ENRICH_JSON_SCHEMA as any } } as any,
      });
      const text = response.content.find(b => b.type === "text")?.text ?? "";
      const parsed = VocabEnrichmentSchema.safeParse(JSON.parse(text));
      if (!parsed.success) { failed++; return; }

      await db.update(vocabularyItems)
        .set({ mnemonic: parsed.data.mnemonic, kanji_breakdown: parsed.data.kanji_breakdown })
        .where(eq(vocabularyItems.id, row.id));
      succeeded++;
    } catch { failed++; }
  }));

  await Promise.all(tasks);
  console.log(`Done. succeeded=${succeeded}, failed=${failed}`);
}
```

### Enrichment Prompt Template

```typescript
function buildEnrichmentPrompt(row: VocabularyItem, hasKanji: boolean): string {
  return `Generate learning aids for this Japanese vocabulary word:

Word: ${row.dictionary_form} (${row.reading}) — "${localize(row.meaning as Localizable, "en")}"
Part of speech: ${row.part_of_speech}
${row.jlpt_level ? `JLPT level: ${row.jlpt_level}` : ""}

Return a JSON object with:

1. "mnemonic": A Localizable object {"en": "...", "pt-BR": "...", "es": "..."}.
   - Each language: one short sentence (10-15 words), playful + visual tone.
   - Use wordplay, vivid imagery, or sound-alikes to the Japanese reading.
   - NOT etymology. NOT anime references.
   - Example EN: "Imagine a KING (OO-SAM-A) sitting on a throne of gold — that's royalty!"

2. "kanji_breakdown": ${hasKanji ? `An object with:
   - "characters": array, one entry per kanji character in "${row.dictionary_form}"
     Each entry: { "char": single kanji, "meaning": Localizable, "on_yomi": kana string,
     "kun_yomi": kana string (empty string if none), "jlpt_level": "N5"/"N4"/"N3"/"N2"/"N1"/null,
     "radical_hint": short string like "water + fish" }
   - "compound_note": Localizable ONLY if the word has 2+ kanji — bridge per-character
     meanings to the compound meaning.
     Example: 鉄道 → "iron (鉄) + road (道) = railway"
` : `null (this word has no kanji characters)`}`;
}
```

### Kanji Breakdown Visual Layout (Claude's Discretion)

**Recommended: stacked compact rows** (matches existing panel rhythm — no card grid, stays inline):

```tsx
// Each row: [kanji] [on'yomi / kun'yomi] [meaning] [radical hint]
function KanjiBreakdownSection({ breakdown, lang }: { breakdown: KanjiBreakdown; lang: string }) {
  return (
    <div>
      <p className="text-xs uppercase tracking-wider text-gray-500 mb-2">Kanji breakdown</p>
      <div className="flex flex-col gap-1.5">
        {breakdown.characters.map((c) => (
          <div key={c.char} className="flex items-baseline gap-2 text-sm">
            <span className="text-lg font-semibold text-white w-6 shrink-0">{c.char}</span>
            <span className="text-gray-400 text-xs">{c.on_yomi}{c.kun_yomi ? ` / ${c.kun_yomi}` : ""}</span>
            <span className="text-gray-300">{localize(c.meaning, lang)}</span>
            <span className="text-gray-600 text-xs ml-auto">{c.radical_hint}</span>
          </div>
        ))}
      </div>
      {breakdown.compound_note && (
        <p className="mt-2 text-xs text-gray-500 italic">{localize(breakdown.compound_note, lang)}</p>
      )}
    </div>
  );
}
```

### Page-Level Data Enrichment

The song page server component is at `src/app/songs/[slug]/page.tsx`. It already fetches the lesson from `song_versions`. This page must additionally query `vocabulary_items` by the UUIDs found in `lesson.vocabulary[*].vocab_item_id` and patch `mnemonic` + `kanji_breakdown` onto each `VocabEntry` before passing the lesson to client components.

This is a targeted join, not a full query rewrite. Pattern: build a Set of `vocab_item_id` UUIDs from the lesson, do a single `WHERE id IN (...)` query on `vocabulary_items`, then merge results by UUID into each `VocabEntry`.

## Migration Mechanics

**Confirmed approach:** Manual SQL file in `drizzle/` directory.

**Reason:** `drizzle-kit generate` runs interactively and has failed before (per project state). The project has a custom migration runner at `scripts/apply-migrations.ts` that processes `.sql` files sorted alphabetically, splits on `--> statement-breakpoint`, handles `already exists` gracefully.

**Column types:**
- `mnemonic jsonb` — stores `{"en": "...", "pt-BR": "...", "es": "..."}` (Localizable shape). JSONB for consistency with `meaning` column on same table.
- `kanji_breakdown jsonb` — stores the structured breakdown object. JSONB (not text + jsonb) for direct Drizzle query capability.

**Nullability:** Both nullable. A NULL `mnemonic` = "not yet enriched". A NULL `kanji_breakdown` = "kana-only word (intentional)". The QA script distinguishes these cases.

**Index:** None needed. Enrichment script query is `WHERE mnemonic IS NULL` on a 705-row table — a sequential scan is faster than an index for small tables.

**Run command:**
```bash
npx tsx scripts/apply-migrations.ts 0004_vocab_enrichment.sql
```

## Open Questions

1. **Model choice for enrichment: Haiku 4.5 vs Sonnet 4.6**
   - What we know: Haiku is cheaper (~10x) and faster. Sonnet produces higher quality creative output.
   - What's unclear: Whether Haiku's mnemonic quality meets the "playful + visual" bar.
   - Recommendation: Start with Haiku. Run enrichment on 10 test rows, eyeball quality. If mnemonics are too literal or generic, switch to Sonnet. The enrichment script should make model configurable via a constant at the top.

2. **Page-level enrichment fetch: N+1 risk**
   - What we know: The lesson has ~10-20 vocab entries. A single `WHERE id IN (...)` query fetches all enrichment data in one round trip.
   - What's unclear: Whether Neon serverless has cold-start latency that makes this perceptible.
   - Recommendation: Single query is fine. If profiling shows latency, consider embedding enrichment data in the lesson JSONB as a cache — but that's premature optimization.

3. **`03-generate-content.ts` prompt update scope**
   - What we know: The script uses Batch API and `03b` uses inline. Both import `buildLessonPrompt` from `scripts/lib/lesson-prompt.ts`.
   - What's unclear: Whether the batch API variant can use `json_schema` output_config with the enriched schema without breaking.
   - Recommendation: Update `buildLessonPrompt` to include the enrichment fields in the schema section. Both `03` and `03b` will pick up the change automatically. Validate by running `--limit 1` on one new song.
