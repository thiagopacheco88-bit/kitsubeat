# Phase 11.3: Fix Untranslated JP Verses — Context

**Inserted:** 2026-04-26
**Status:** Draft (pending `/gsd-spec-phase` refinement)
**Surfaced by:** Sign-flow verse 8 「忘れてしまえばいいよ 感じなくなっちゃえばいい」 rendered as `(untranslated)` on Vercel; catalog audit then revealed the same pattern across 102 of 274 songs.

<product_lens>
## Product lens (why this phase exists)

KitsuBeat's lesson UI shows a Japanese line, the user's-language translation, and a per-token breakdown. When a verse renders `(untranslated)` mid-song, the user loses the comprehension thread and the streak motivation degrades — the same streak-killer dynamic from Phase 11.2 but at the content layer instead of timing.

Catalog scope (audit 2026-04-26):
- **102 of 274 songs (37%)** have at least one untranslated JP verse.
- **970 broken verses total** across the catalog.
- **Worst offenders**: again-yui (38), whats-up-people-maximum-the-hormone (29), diver-nico-touches-the-walls (28), mezamero-yasei-matchy-with-question (22), name-of-love-cinema-staff (21), yellow-moon-akeboshi (21).
</product_lens>

<root_cause>
## Likely root cause

Likely a side effect of commit bec890c (2026-04-18, Phase 08.4 closeout):
> "lesson-prompt §1: add explicit coverage requirement — every Japanese lyric line must map to a verse"

The new prompt forced the LLM to generate verse entries for previously-skipped filler lines, but for some lines it cheaped out — instead of tokenising and translating, it emitted a single-token "other" entry with `surface = full line including 「」`, `meaning.en = "(untranslated lyric line)"`, and verse-level `translations = { en: "(untranslated)", ... }`.

The Apr 26 "lesson cache normalization" commit (4b80877) may have additionally reshaped legacy entries into this stub form via [transform-lesson-shape.ts](../../../scripts/seed/transform-lesson-shape.ts).

Detection signature for an untranslated-stub verse:
```ts
v.tokens.length === 1 &&
hasJapanese(v.tokens[0].surface) &&
v.translations.en.startsWith("(")
```

Verses with `translations.en === "(untranslated)"` AND tokens that contain only English are legitimate (e.g. "Can you hear me?", "So am I" — no JP to translate). The detection above filters those out correctly.
</root_cause>

<solution_shape>
## Solution shape

### Approach: local LLM (Ollama) drafts + inline review

User constraint (memory): no paid API credits. So inline Claude Code (the existing lesson-generation pattern) is one option but expensive in attention. Local LLM is the primary path.

### Pipeline

1. **Audit and queue** ([scripts/seed/audit-untranslated-verses.ts](../../../scripts/seed/audit-untranslated-verses.ts), to build): scan `data/lessons-cache/*.json`, detect stub verses, write `.planning/translation-queue.tsv` with `slug \t verse_number \t surface`.

2. **Local-LLM draft** ([scripts/seed/draft-untranslated-translations.ts](../../../scripts/seed/draft-untranslated-translations.ts), to build): for each queue entry, prompt Ollama with:
   - The full song's raw lyrics (whole-song context — the user's specific request)
   - The 2 verses immediately before and after the broken verse, with their existing translations (anchors for tone consistency)
   - A schema-locked output spec matching the existing token shape (kanji/reading/romaji/grammar/grammar_color/meaning per token + verse-level translations in en/pt-BR/es)
   - Model recommendation: `qwen2.5:14b-instruct` (strong JP) or `llama3.1:8b` (lighter) — pick after a 10-verse pilot to compare quality.
   - Output: write JSON patch into [.planning/verse-patches/{slug}.json](../../../.planning/verse-patches/) using the existing schema consumed by [apply-verse-patch.ts](../../../scripts/seed/apply-verse-patch.ts) — but with `replace_verse_number` semantics added (existing patch infra is append-only; needs a small extension).

3. **Inline review for high-difficulty verses**: queue verses where Ollama confidence is low (e.g. tokens with `meaning.en` containing "?", or schema violations) for inline-Claude-Code review. Estimated 10-20% of the 970 — the rest can ship from Ollama drafts.

4. **Splice + push**: extend [apply-verse-patch.ts](../../../scripts/seed/apply-verse-patch.ts) to support `replace_verse_number` (current code only inserts `after_original`). Run patch apply per slug, then `npx tsx scripts/seed/05-insert-db.ts --slug=<slug>` (slug filter shipped 2026-04-26 in commit `7f1dc95`).

### Throughput estimate
- Audit + queue: minutes.
- Ollama drafts on a CPU laptop: ~30-60 seconds per verse × 970 verses = ~12-16 hours total. Run in batches via [run-pending-batch.sh](../../../scripts/seed/run-pending-batch.sh)-style orchestration overnight.
- Inline review: 10-20% × 970 ≈ 100-200 verses. Two or three sessions of work.
- DB push (slug-batched): minutes per song × 102 songs ≈ 1-2 hours.

</solution_shape>

<scope_decisions>
## In scope
- Audit script that detects stub verses (signature above)
- Ollama-based draft pipeline with whole-song + neighbour-verse context
- Extension to apply-verse-patch.ts for `replace_verse_number` semantics
- Inline review path for low-confidence drafts
- Catalog-wide rollout via slug-batched 05-insert-db pushes

## Out of scope
- Improving the lesson-prompt itself ([scripts/lib/lesson-prompt.ts](../../../scripts/lib/lesson-prompt.ts) is fine; the bug is that the LLM cheaped out on a subset of lines, not that the prompt is broken)
- Re-generating entire lessons (too expensive; spot-fix only the broken verses)
- Translation quality grading framework (separate concern; if Ollama drafts feel weak, fall back to inline review per-verse)
</scope_decisions>
