**Step A — `src/lib/review/queue-builder.ts` (pure, testable):**

Define the queue question types up front — only three are allowed (CONTEXT: Fill-the-Lyric excluded).

```ts
import type { ExerciseType } from "@/lib/exercises/generator";

export type ReviewQuestionType = Exclude<
  ExerciseType,
  "fill_lyric"
>; // {vocab_meaning | meaning_vocab | reading_match}

export interface DueCardInput {
  vocab_item_id: string;
  state: 0 | 1 | 2 | 3;
  due: Date;
}
export interface NewCardInput {
  vocab_item_id: string;
}
export interface ReviewQueueItem {
  vocab_item_id: string;
  exerciseType: ReviewQuestionType;
  isNew: boolean;
}

/**
 * Combines due + new cards into a session queue.
 * - Due cards first (ordered by `due ASC` upstream — preserve order here).
 * - New cards appended, bounded by `newCardBudget`.
 * - Each card gets assigned a single exercise type chosen deterministically from
 *   {vocab_meaning, meaning_vocab, reading_match} based on the vocab_item_id hash,
 *   so repeated reviews of the same card rotate types over time.
 * - Fill-the-lyric is NEVER emitted (CONTEXT-locked).
 */
export function buildReviewQueue(
  due: DueCardInput[],
  newCards: NewCardInput[],
  newCardBudget: number
): ReviewQueueItem[];
```

Implementation sketch:
- Type rotation: `const types: ReviewQuestionType[] = ["vocab_meaning", "meaning_vocab", "reading_match"]`. For each card, compute `types[hash(vocab_item_id) % 3]`. Simple `Array.from(vocab_item_id).reduce((acc,c) => (acc*31 + c.charCodeAt(0)) | 0, 0)` hash is fine.
- Return due cards first (their input order), then `newCards.slice(0, Math.max(0, newCardBudget))`.
- If `newCardBudget <= 0`, drop all new cards.

**Step A — `src/app/actions/review.ts` ("use server"):**

```ts
"use server";
import { sql } from "drizzle-orm";
import { db } from "@/lib/db";
import { recordVocabAnswer } from "@/app/actions/exercises";
import { isPremium } from "@/app/actions/userPrefs";
import { REVIEW_NEW_DAILY_CAP } from "@/lib/user-prefs";
import type { ExerciseType } from "@/lib/exercises/generator";

function todayIsoDate(): string {
  return new Date().toISOString().slice(0, 10); // "YYYY-MM-DD"
}

export async function consumeNewCardBudget(
  userId: string
): Promise<{ allowed: boolean; remaining: number }> {
  const today = todayIsoDate();
  const cap = REVIEW_NEW_DAILY_CAP;
  // Atomic upsert with rollover. See RESEARCH §Code Examples — daily counter.
  const rows = await db.execute<{ review_new_today: number }>(sql`
    INSERT INTO users (id, review_new_today, review_new_today_date)
    VALUES (${userId}, 1, ${today}::date)
    ON CONFLICT (id) DO UPDATE SET
      review_new_today = CASE
        WHEN users.review_new_today_date = ${today}::date
             AND users.review_new_today >= ${cap} THEN users.review_new_today
        WHEN users.review_new_today_date = ${today}::date
          THEN users.review_new_today + 1
        ELSE 1
      END,
      review_new_today_date = ${today}::date,
      updated_at = NOW()
    RETURNING review_new_today
  `);
  const raw = Array.isArray(rows) ? rows : (rows.rows ?? []);
  const count = Number(raw[0]?.review_new_today ?? 0);
  return { allowed: count <= cap, remaining: Math.max(0, cap - count) };
}

export async function startReviewSession(userId: string) {
  const premium = await isPremium(userId);
  if (!premium) return { gated: true as const, reason: "premium_required" };
  // The actual queue is built by the /api/review/queue endpoint client-side;
  // this server action is the narrow premium check a client can call to decide
  // whether to show upsell. Kept minimal — no queue work here, avoids duplicate
  // reads if the client is going to call /api/review/queue anyway.
  return { gated: false as const };
}

export async function recordReviewAnswer(input: {
  userId: string;
  vocabItemId: string;
  exerciseType: Exclude<ExerciseType, "fill_lyric">;
  correct: boolean;
  revealedReading?: boolean;
  responseTimeMs: number;
  isNew: boolean;
}) {
  const premium = await isPremium(input.userId);
  if (!premium) throw new Error("premium_required");

  // New-card gate: if it's a "new" card, burn a budget slot before recording.
  // If the budget is already spent today, reject — UI should stop serving new
  // cards once the budget hits zero, but a race check here is the source of truth.
  if (input.isNew) {
    const budget = await consumeNewCardBudget(input.userId);
    if (!budget.allowed) {
      throw new Error("daily_new_card_cap_reached");
    }
  }

  // Linearize writes — neon-http has no callback transactions (Pitfall 4).
  // recordVocabAnswer already does FSRS + exercise_log in sequence;
  // songVersionId=null is explicitly supported.
  return recordVocabAnswer({
    userId: input.userId,
    vocabItemId: input.vocabItemId,
    songVersionId: null,
    exerciseType: input.exerciseType,
    correct: input.correct,
    revealedReading: input.revealedReading,
    responseTimeMs: input.responseTimeMs,
  });
}
```

```tsx
export const dynamic = "force-dynamic";

import { sql } from "drizzle-orm";
import { db } from "@/lib/db";
import { isPremium } from "@/app/actions/userPrefs";
import { REVIEW_NEW_DAILY_CAP } from "@/lib/user-prefs";
import ReviewLanding from "./ReviewLanding";

const PLACEHOLDER_USER_ID = "test-user-e2e";

async function countDue(userId: string): Promise<number> {
  const rows = await db.execute<{ count: number }>(sql`
    SELECT COUNT(*)::int AS count
    FROM user_vocab_mastery
    WHERE user_id = ${userId}
      AND state IN (1, 2, 3)
      AND due <= NOW()
  `);
  const raw = Array.isArray(rows) ? rows : (rows.rows ?? []);
  return Number(raw[0]?.count ?? 0);
}

async function readTodayBudget(userId: string): Promise<number> {
  const today = new Date().toISOString().slice(0, 10);
  const rows = await db.execute<{ review_new_today: number; review_new_today_date: string | null }>(sql`
    SELECT review_new_today, review_new_today_date::text AS review_new_today_date
    FROM users WHERE id = ${userId}
  `);
  const raw = Array.isArray(rows) ? rows : (rows.rows ?? []);
  const row = raw[0];
  if (!row) return REVIEW_NEW_DAILY_CAP;
  if (row.review_new_today_date !== today) return REVIEW_NEW_DAILY_CAP;
  return Math.max(0, REVIEW_NEW_DAILY_CAP - row.review_new_today);
}

export default async function ReviewPage() {
  const userId = PLACEHOLDER_USER_ID;
  const [premium, dueCount, newBudget] = await Promise.all([
    isPremium(userId),
    countDue(userId),
    readTodayBudget(userId),
  ]);

  return (
    <ReviewLanding
      isPremium={premium}
      dueCount={dueCount}
      newBudgetRemaining={newBudget}
      dailyCap={REVIEW_NEW_DAILY_CAP}
    />
  );
}
```

**Race-handling for daily-cap (M4 fix) — MANDATORY step in ReviewSession.tsx:**

The queue is assembled on the server using `budget_remaining` at queue-fetch time, but per-card `consumeNewCardBudget` upserts run later. Two-tab scenarios, stale queues held across browser restarts, or even a long idle mid-session can cause `recordReviewAnswer` to throw `"daily_new_card_cap_reached"` mid-flight. Handle it:

- Error-code contract: `recordReviewAnswer` already throws `new Error("daily_new_card_cap_reached")` when the budget is exhausted. The client MUST match on `err.message === "daily_new_card_cap_reached"`.
- When that specific error fires on a card marked `isNew: true`:
  1. Do NOT mark the card answered in the review store. Instead, call `advance()` to skip it AND remove any remaining `isNew: true` items from the queue (`useReviewSession.getState().items.filter(i => !i.isNew)` followed by `load(filtered)` and `advance()` — or add a `removeNewCards()` action to the store in Task 2).
  2. Refetch `budget_remaining` from the server via a new lightweight endpoint `GET /api/review/budget` (see next section) — this confirms the server's view of the counter for the user's UI.
  3. Render a non-blocking toast banner at the top of the session: "Daily new-card limit reached — continuing with review-only cards for today". Use a minimal Tailwind `<div role="status" aria-live="polite">` — no new library.
  4. If no `isNew: false` (due) cards remain either, end the session and show the summary screen.
- All other throws (network, generic server error) bubble up with a standard "Couldn't record answer — try again" inline message; no queue mutation.

**Also update Task 2's Zustand store spec:** add a `removeNewCards()` action that returns `{ ...state, items: state.items.filter(i => !i.isNew) }` so ReviewSession can call it cleanly.
