**Step B — five query functions appended to `src/lib/db/queries.ts`:**
Use the `db.execute(sql\`...\`)` escape hatch — same pattern as `saveSessionResults` step 8. Each function must handle both `rows` array shapes (`Array.isArray(r) ? r : (r.rows ?? [])` per existing queries).

1. `export async function getKnownWordCountForSong(userId: string, songId: string): Promise<{ total: number; known: number; mastered: number; learning: number }>`
   - CTE: `WITH song_vocab AS (SELECT DISTINCT vg.vocab_item_id FROM vocab_global vg WHERE vg.song_id = ${songId}::uuid)` then `LEFT JOIN user_vocab_mastery m ON m.vocab_item_id = s.vocab_item_id AND m.user_id = ${userId}`.
   - `known = COUNT(*) FILTER (WHERE m.state IN (1,2,3))` — Pitfall 1 in RESEARCH.md: Tier 2+ = states 1,2,3 (NOT raw state >= 2).
   - `mastered = COUNT(*) FILTER (WHERE m.state = 2)`, `learning = COUNT(*) FILTER (WHERE m.state IN (1,3))`.
   - `total = COUNT(*)` over song_vocab (already DISTINCT to avoid tv/full double-count — Pitfall 2).
   - Return `{ total, known, mastered, learning }` all as `Number(r?.field ?? 0)`.

2. `export async function getGlobalLearnedCount(userId: string): Promise<number>`
   - `SELECT COUNT(*)::int AS count FROM user_vocab_mastery WHERE user_id = ${userId} AND state IN (1, 2, 3)`. No JOIN to vocab_global needed — mastery rows are already keyed by vocab_item_id.
   - Return `Number(raw[0]?.count ?? 0)`.

3. `export async function getSeenInSongsForVocab(vocabItemId: string): Promise<Array<{ slug: string; title: string; anime: string }>>`
   - `SELECT DISTINCT s.slug, s.title, s.anime FROM vocab_global vg JOIN songs s ON s.id = vg.song_id WHERE vg.vocab_item_id = ${vocabItemId}::uuid ORDER BY s.title ASC`.
   - DISTINCT is mandatory because `vocab_global` has one row per (vocab_item_id, song_id, version_type) — without DISTINCT a song with both tv+full versions would appear twice (Pitfall 2).

4. `export async function getVocabularyDashboard(userId: string, opts: { tierFilter?: 1 | 2 | 3; sourceSongId?: string; limit?: number; sortDirection?: "asc" | "desc" } = {}): Promise<DashboardRow[]>`
   - Define an exported `interface DashboardRow { vocab_item_id: string; dictionary_form: string; reading: string; romaji: string; meaning: unknown; part_of_speech: string; jlpt_level: string | null; state: 0 | 1 | 2 | 3; due: Date; last_review: Date | null; reps: number; source_song_count: number }` in the same file.
   - **Tier → state mapping (LOCKED — Path B, phase-local 3-bucket split):**
     - `tierFilter === 3` (Mastered) → `AND m.state = 2`
     - `tierFilter === 2` (Known)    → `AND m.state = 3`
     - `tierFilter === 1` (Learning) → `AND m.state = 1`
     - `tierFilter` omitted           → no extra state clause beyond the base WHERE
   - **Divergence note — MANDATORY to preserve in code comments AND in the SUMMARY:**
     The dashboard uses a phase-local 3-bucket mapping that diverges from `src/lib/fsrs/tier.ts`, which collapses to only 2 post-new buckets (`state IN (1,3) → TIER_LEARNING`, `state=2 → TIER_REVIEW`). Rationale: the dashboard is the only surface where users distinguish "seen and relearning" (state=3) from "fresh learning" (state=1), so a richer split is warranted here. All other surfaces (TierText, KnownWordCount, pills) continue to use `tierFor()` unchanged — do NOT silently redefine `tierFor`. The single-character mismatch (tier.ts treats state=1 and state=3 identically; the dashboard does not) is deliberate and must be documented in the JSDoc above `getVocabularyDashboard`.
   - sourceSongId filter uses `AND EXISTS (SELECT 1 FROM vocab_global vg2 WHERE vg2.vocab_item_id = m.vocab_item_id AND vg2.song_id = ${opts.sourceSongId}::uuid)`.
   - Base WHERE: `m.user_id = ${userId} AND m.state IN (1, 2, 3)` (never surface pure-new `state=0` cards — they are filtered out by the WHERE clause, which is why `tierFilter=0`/`tier.ts` TIER_NEW is not represented in this function's filter surface).
   - SELECT includes `(SELECT COUNT(DISTINCT vg.song_id)::int FROM vocab_global vg WHERE vg.vocab_item_id = m.vocab_item_id) AS source_song_count` for the expandable "Seen in" count.
   - ORDER BY: `m.state ${orderDir}, m.last_review DESC NULLS LAST` — Claude discretion (Open Question #1 in RESEARCH.md): surfaces mastery wins first, then recency within each tier. Documented inline. This ordering also acts as the free-tier preview cutoff when `opts.limit=20`.
   - LIMIT clause appends only when `opts.limit` is provided.
   - JOIN `vocabulary_items vi ON vi.id = m.vocab_item_id` for the word metadata (dictionary_form, reading, romaji, meaning, part_of_speech, jlpt_level).

5. `export async function getDueReviewQueue(userId: string, newCardCap: number, now: Date = new Date()): Promise<{ due: Array<{ vocab_item_id: string; state: 0|1|2|3; due: Date }>; new: Array<{ vocab_item_id: string }> }>`
   - Query 1 (due): `SELECT m.vocab_item_id, m.state, m.due FROM user_vocab_mastery m WHERE m.user_id = ${userId} AND m.state IN (1, 2, 3) AND m.due <= ${now.toISOString()}::timestamptz ORDER BY m.due ASC`. Uncapped per CONTEXT.
   - Query 2 (new): `SELECT DISTINCT vg.vocab_item_id FROM vocab_global vg WHERE NOT EXISTS (SELECT 1 FROM user_vocab_mastery m WHERE m.user_id = ${userId} AND m.vocab_item_id = vg.vocab_item_id) ORDER BY vg.vocab_item_id LIMIT ${newCardCap}`. Bounded by `newCardCap` (Plan 05 will compute this from REVIEW_NEW_DAILY_CAP minus already-consumed).
   - Two separate `db.execute` calls — neon-http has no callback transactions (Pitfall 4); sequential is fine for reads.

Add JSDoc above each function noting which requirement ID it services (CROSS-01, CROSS-02, etc.). No wiring to any route/component — that's the subsequent plans' job.
