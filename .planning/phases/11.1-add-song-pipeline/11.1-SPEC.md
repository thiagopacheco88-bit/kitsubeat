# Phase 11.1: Add-Song Pipeline — Specification

**Created:** 2026-04-26
**Ambiguity score:** 0.18 (gate: ≤ 0.20)
**Requirements:** 3 locked

## Goal

Close three concrete gaps in the existing add-song pipeline so adding a new song to KitsuBeat is reliable enough that no song ships a streak-killer (broken karaoke, half-written DB row, duplicate YouTube ID), and operator overhead is reduced via a single-page README. The pipeline itself (the 8 gates documented in `docs/SONG_INGESTION_SOP.md`) is not rewritten — only the three gaps are filled.

## Background

KitsuBeat's add-song pipeline is ~80% built. The 8-gate SOP (curation → YouTube selection → lyrics → quality validation → WhisperX timing → TV derivation → lesson generation/audit → DB insert) is documented in 230 lines at `docs/SONG_INGESTION_SOP.md` and composed by two batch shells (`run-pending-batch.sh`, `overnight-orchestrator.sh`). 273 songs are in the Neon DB via this pipeline. A 2026-04-26 audit against the original 11-validation CONTEXT.md found three remaining gaps:

1. **No cross-song `youtube_id` uniqueness check.** During the TV backfill, 5 `song_versions` rows shared `gX9WX_NP5Ug` — the same YouTube video claiming to be 5 different songs. Recovered ad-hoc; no automated guard exists. Failure mode: user picks song A, gets song B's audio.

2. **`05-insert-db.ts` is non-transactional.** Per-song upserts of `songs` and `song_versions` happen as separate statements. If the second write fails (network blip, schema drift), the catalog has a song row with no playable version — `/songs` lists it but the click leads to a broken page. Beginner's first impression: failure.

3. **No operator quick-start.** New Claude Code sessions re-derive the gate ordering from the 230-line SOP every time, costing tokens and time. The SOP is a reference doc, not a quick-start.

**Why this matters now (product lens):** KitsuBeat's product DNA is addictive entertainment — consistency over precision. Each gap above is a streak-killer or onboarding-killer. Catalog velocity (Phase 12 learning path needs a deep growing catalog) is bottlenecked partly by operator overhead and reliability.

## Requirements

1. **Cross-song youtube_id audit**: A standalone script flags any duplicate `youtube_id` across `song_versions` and exits non-zero on detection.
   - Current: No `audit-yt-ids.ts` exists. Cross-song uniqueness is unenforced. The TV backfill's 5-row dupe was caught by manual inspection only.
   - Target: `npx tsx scripts/seed/audit-yt-ids.ts` queries `song_versions`, groups by `youtube_id`, and reports any group with `count > 1`. Output is CSV with columns `youtube_id, count, slugs, version_types` written to `data/yt-id-audit.csv`. On any duplicate, prints summary to stderr and exits with code 1. On clean run, prints "OK: N rows checked, 0 duplicates" to stdout and exits 0. Wired into both `run-pending-batch.sh` and `overnight-orchestrator.sh` as the final step (after DB insert) so any batch that introduces a duplicate halts loudly.
   - Acceptance: Test fixture inserts a known dupe → script exits 1 and the CSV contains the duplicate row. Removing the dupe → script exits 0. Both batch shells fail loudly when the audit detects duplicates.

2. **Per-song transactional DB write**: `05-insert-db.ts` wraps each song's `songs` + `song_versions` upserts in a single transaction; partial failures roll back that song without affecting others in the batch.
   - Current: `05-insert-db.ts` per-song flow is `db.insert(songs).onConflictDoUpdate(...)` followed by `db.insert(songVersions).onConflictDoUpdate(...)` as two separate statements. A failure between them leaves a `songs` row without its `song_versions` row → broken `/songs/<slug>` page.
   - Target: Per-song upsert wrapped in `db.transaction(async (tx) => { ... })`. Songs row writes via `tx`; both song_versions writes (`full` and `tv` when present) write via `tx`. Any thrown error rolls back that song's transaction; the outer batch loop catches, logs the slug + error, and continues with the next song. Verified on Neon's HTTP serverless driver — switch to the WebSocket / pooled driver if HTTP doesn't support multi-statement transactions.
   - Acceptance: Simulating a `song_versions` write failure (e.g. forced FK violation or mocked exception) leaves zero rows in `songs` for that slug — confirmed by `SELECT FROM songs WHERE slug = 'test-rollback'` returning 0 rows. Successful insert leaves both `songs` AND `song_versions` rows. A batch run with one failing song and one passing song results in exactly the passing song's rows being written.

3. **Operator quick-start README**: A single-page reference at `docs/README-adding-songs.md` directs operators to the right batch shell per scenario, with no decision-tree larger than 5 nodes deep.
   - Current: Operators (and Claude Code sessions) re-read the 230-line `docs/SONG_INGESTION_SOP.md` from scratch each session to derive the gate order, costing context tokens and onboarding time. There is no "I want to add 3 songs — what command do I run?" entry point.
   - Target: `docs/README-adding-songs.md` exists and contains: (a) a one-paragraph "what this pipeline does" intro; (b) a 5-row scenario table mapping common operator intents to the exact command(s) to run; (c) a "before you start" checklist (Python venv path, Node version, free RAM for WhisperX); (d) explicit links to the SOP for each gate. Total length ≤ 100 lines so it loads cheaply at session start.
   - Acceptance: README exists at `docs/README-adding-songs.md`. The 5 scenarios covered are: "Add 1 new song", "Add a batch of N songs", "Resume pending_whisper backlog", "Add only the TV cut for an existing song", "Replace a geo-blocked / live-version YouTube ID". Each row has a runnable command. File is ≤ 100 lines. Markdown lint passes (no broken links to SOP gate sections).

## Boundaries

**In scope:**
- New script: `scripts/seed/audit-yt-ids.ts` (Requirement 1)
- Wiring of audit into `run-pending-batch.sh` and `overnight-orchestrator.sh` as the final step (Requirement 1)
- Modification of `scripts/seed/05-insert-db.ts` to wrap per-song upserts in `db.transaction()` (Requirement 2)
- Verification (and migration if needed) of Neon driver transaction support (Requirement 2)
- New doc: `docs/README-adding-songs.md` (Requirement 3)

**Out of scope:**
- **Rewriting the 8-gate SOP or any of the seed scripts beyond `05-insert-db.ts`** — the pipeline works; this phase composes / closes gaps only.
- **`add-songs.ts` orchestrator (Gap 3 option B)** — explicitly deferred. The SOP + existing batch shells are sufficient. Catalog velocity is bottlenecked by Claude session time for inline lesson generation, not by orchestration tooling. Revisit only if velocity metrics show otherwise.
- **Pre-commit git hooks or CI workflows** — audit runs post-batch and on demand only. No greenfield CI infra in this phase.
- **Whole-batch transactions** — explicitly rejected; bad fit for overnight pipelines where partial progress beats zero progress.
- **Wrapping `vocabulary_items` writes in the same transaction** — those live in a separate vocab-backfill flow (Phase 7) and coupling them here would over-scope the change.
- **Lesson regeneration or quality fixes** — Phase 11.1 is about *adding* songs, not improving existing lessons.
- **Admin web UI** — CLI only.
- **Cover/remix detection improvements** — current LCS-fail rejection is sufficient.
- **Pruning the 50 manifest entries with no cached lesson** — separate cleanup task, not blocking.

## Constraints

- **Drizzle + Neon serverless driver:** Transaction semantics must be verified before declaring Requirement 2 complete. If the HTTP driver does not support `db.transaction()` for the songs+song_versions write pattern, switch the script to the WebSocket / unpooled driver (already used elsewhere in the repo per `feat(db): add cold-start retry wrapper via neonConfig.fetchFunction`).
- **Audit script must be read-only against the database.** No DELETE, UPDATE, or schema changes. Pure SELECT + report.
- **README length budget:** ≤ 100 lines. Larger docs don't load cheaply at session start and defeat the purpose.
- **No new external dependencies.** Audit script and README use existing project setup; no new npm packages.
- **Audit CSV path:** `data/yt-id-audit.csv` — added to `.gitignore` (it's ephemeral data, not source).
- **Both batch shells (`run-pending-batch.sh`, `overnight-orchestrator.sh`) must call the audit at end-of-batch.** A single missed wiring defeats the guard.
- **Transactional rollback must NOT swallow errors** — failures log to stderr with slug + error message so operators see what failed.

## Acceptance Criteria

- [ ] `scripts/seed/audit-yt-ids.ts` exists and runs via `npx tsx scripts/seed/audit-yt-ids.ts`
- [ ] Test fixture with a known dupe `youtube_id` causes the audit to exit non-zero and write the dupe to `data/yt-id-audit.csv`
- [ ] Test fixture with no dupes causes the audit to exit 0 and print "OK: N rows checked, 0 duplicates"
- [ ] `run-pending-batch.sh` invokes `audit-yt-ids.ts` after the DB-insert step and propagates non-zero exit codes
- [ ] `overnight-orchestrator.sh` invokes `audit-yt-ids.ts` after the DB-insert step and propagates non-zero exit codes
- [ ] `scripts/seed/05-insert-db.ts` per-song flow uses `db.transaction()` wrapping `songs` + `song_versions` upserts
- [ ] Forced failure mid-transaction (mocked or simulated FK violation) leaves zero rows for that slug in both `songs` and `song_versions`
- [ ] Successful insert leaves matching rows in both `songs` and `song_versions`
- [ ] Mixed batch (one failing, one passing) writes only the passing song's rows; failing slug logs to stderr; batch exits non-zero only if every song failed (otherwise exits 0)
- [ ] `docs/README-adding-songs.md` exists, is ≤ 100 lines, and covers all 5 named scenarios with runnable commands
- [ ] All links from `docs/README-adding-songs.md` to `docs/SONG_INGESTION_SOP.md` gate sections resolve
