# Phase 08.2: FSRS Progressive Disclosure - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire the currently-dead `user_vocab_mastery` and `user_exercise_log` tables (schema landed in Phase 7, never written to) so Phase 8 exercise activity persists as per-vocab FSRS state. Use that mastery signal to derive a three-tier display for each word — Tier 1 (kanji + furigana + romaji), Tier 2 (kanji + furigana), Tier 3 (kanji only) — so exercise rendering adapts to what the user has already mastered, instead of every option starting at bare kanji.

Scoped to the exercise flow only. Cross-song aggregation dashboards, premium review queues, and global vocab counters are **Phase 11**. Standalone kana trainer is **Phase 9**. Listening/sentence-order exercises are **Phase 10**. Player lyrics and the song-page vocab section keep their existing global toggles untouched in this phase.

</domain>

<decisions>
## Implementation Decisions

### Write triggers & events
- **`user_vocab_mastery` writes: per-answer, inline.** Each correct/wrong answer immediately updates FSRS state, stability, difficulty, due, reps, last_review for that word — same-request commit, no end-of-session batching. Matches Anki/FSRS semantics; a mid-session browser close does not lose work.
- **`user_exercise_log` writes: one row per question attempt.** Fine-grained: exercise_type, vocab_id, correct/wrong, response_time (plus whatever scalar columns the Phase 7 schema defines). High volume is acceptable; we want later analytics on wrong-answer patterns.
- **Exercise contribution is weighted, not uniform.** All four Phase 8 exercise types (vocab→meaning, meaning→vocab, reading-match, fill-lyric) feed FSRS, but with different review-rating weights. Production-flavored types (meaning→vocab) weigh more than pure-surface types (reading-match). **Claude's Discretion** on the exact weights — pick defaults that map cleanly onto FSRS rating inputs (Again / Hard / Good / Easy) and document them in one place.
- **Distractors do NOT count as exposure.** Only the target word (the one being asked about or blanked) receives an FSRS update on a given question. Distractor words Y/Z in a fill-lyric question where blank=X get no mastery or "seen" touch. Keeps the mastery signal clean — we don't pollute a word's stability just because it flashed by as a wrong option.

### Tier algorithm & thresholds
- **Tier signal = FSRS state, mapped directly.**
  - `state = new` → Tier 1 (kanji + furigana + romaji)
  - `state = learning` → Tier 2 (kanji + furigana)
  - `state = review` → Tier 3 (kanji only)
  - `state = relearning` → treat as `learning` for tier purposes (Tier 2).
- **Stability thresholds are not used directly — FSRS's own state transitions drive the tier.** Transition timing uses FSRS defaults (request_retention ≈ 0.9, default initial stability, etc.); no custom cutoff constants.
- **Brand-new words always start at Tier 1** — full assistance on every first exposure, without exception. No JLPT-level-based shortcut, no kanji-vs-kana branching at cold start. Pure-kana words at Tier 1 degrade gracefully (nothing to render as "furigana" — just the kana itself).
- **Tier is recomputed per question, not snapshotted per session.** When a user misses a word, that word's FSRS state updates immediately (per the inline write rule above), so if the same word reappears later in the session — or in its own feedback panel — the tier already reflects the regression. Accepts some mid-session "flicker" in exchange for responsiveness and a single source of truth.

### Where tiers apply
- **Tiers affect the exercise flow only** — prompt text, answer options, and feedback panel. The Phase 2 player lyrics and the song-page vocabulary breakdown section are out of scope for this phase and keep their existing global furigana/romaji toggle behavior unchanged.
- **Exercise-type semantics override tier when they conflict.** Reading-match prompts and meaning→vocab answer options must never show furigana/romaji on the answer-revealing surface, regardless of what tier the word is at — showing it would leak the answer. Tier still governs everywhere it doesn't leak. The post-answer feedback panel always shows full detail (see below), so the withheld info surfaces right after the attempt.
- **Feedback panel is always Tier 1, ignoring per-word tier.** After every answer (correct or wrong), the panel shows kanji + furigana + romaji + meaning for the target word. Feedback is the teaching moment; we never withhold information there, even for Tier 3 mastered words.
- **Answer options honor tier when they contain kanji,** subject to the exercise-semantic override above.

### Backfill & visibility
- **Start cold — no backfill of existing Phase 8 progress.** Existing users who already have `user_song_progress` rows get zero `user_vocab_mastery` rows at deploy time. Every word reads as `new` → Tier 1 until they play a post-deploy session. Accepted tradeoff: cleanest semantics, no guesswork about what "completed a session" means in FSRS terms.
- **User-visible mastery details surface on tap/hover.** In the feedback panel and session summary, tapping a word reveals reps, correct %, state, and next-due date. No passive badges, dots, or mastery indicators in the question flow — kept out of the hot path. Power-user transparency on demand.
- **Per-question "reveal reading" escape hatch.** On a question, the user can tap to reveal furigana/meaning if stuck. The reveal counts as a non-correct answer for FSRS purposes (at minimum does not award a "correct" rating; treat it as equivalent to a wrong answer or "Again" rating — **Claude's Discretion** on the exact FSRS rating mapping). No global "always show furigana" settings toggle — we don't want an escape hatch that silently undoes the phase's value.
- **Global player furigana/romaji toggles stay independent of exercise tier.** Toggle state controls player lyrics and vocab section only (unchanged Phase 2 behavior). It neither overrides nor is overridden by the per-word tier in the exercise flow. Two independent rendering systems for two independent surfaces.

### Claude's Discretion
- Exact FSRS review-rating weights per exercise type (how each of the 4 exercises maps onto FSRS's Again / Hard / Good / Easy). Define in one config module so weights are tunable later.
- The FSRS rating applied when the user taps the "reveal reading" escape hatch (likely Again, but may be Hard — pick whichever keeps the word in shorter rotation without over-penalizing).
- Exact structure of the tap-to-inspect mastery details panel (fields, layout).
- How `state = relearning` gets surfaced visually (treated as `learning` / Tier 2, but ordering/styling in session summary is open).
- DB write batching at the transport layer (e.g., whether the per-answer writes go through a shared optimistic mutation helper) — as long as the user-visible semantics are "persisted on answer," the implementation is flexible.
- Handling of pure-kana words in the tier system (all three tiers render identically for them; no special-case code required, but note in the renderer so future readers aren't confused).

</decisions>

<specifics>
## Specific Ideas

- Tier 1/2/3 correspond exactly to the roadmap's "kanji + furigana + romaji → kanji + furigana → kanji only" progression — three tiers, no more, no fewer.
- Behavior model is "Anki-like": per-answer commits, FSRS state drives difficulty, user never sees the card scheduler machinery unless they explicitly tap to inspect.
- Reveal-reading escape hatch should feel closer to "I peeked" than "give up" — the user continues the question, picks an answer, and the reveal just biases the FSRS rating downward.
- The tap-to-inspect mastery detail is a power-user affordance (like hovering an Anki card's info panel) — not something shoved in the user's face.

</specifics>

<deferred>
## Deferred Ideas

- **Cross-song vocab dashboard, "You know 8/12 words in this song" counts, and the premium SRS review queue.** These are Phase 11 — they depend on per-vocab FSRS mastery being persisted, which is exactly what this phase delivers. Do not build them here.
- **Per-word tier on the song page's vocabulary breakdown section.** User chose "exercises only" for rollout scope. A future phase (likely tied to Phase 11's cross-song surfacing) can extend tier-aware rendering to the vocab section.
- **Per-word tier on the player lyrics line itself.** Same reasoning — deliberately excluded from this phase to keep blast radius small and to preserve the existing global furigana/romaji toggle contract.
- **Global "always show furigana" user setting.** Considered and rejected for this phase — it would let users opt out of the disclosure system. If a product reason to offer it emerges later, revisit.
- **Proxy backfill of mastery from `user_song_progress` history.** Considered and rejected; a future migration phase could replay completed-session data into synthetic FSRS updates if cold-start UX turns out to be painful in practice.
- **JLPT-level-adaptive starting tier.** Considered for cold start ("N5 words for an N5+ user could skip Tier 1") and rejected for now — depends on a per-user JLPT level signal that doesn't exist yet.
- **Listening drill, sentence order, and grammar conjugation exercises writing to FSRS.** These exercise types belong to Phase 10; this phase only wires the four existing Phase 8 exercise types.
- **Session-start "mark due" filter that only pools FSRS-due words.** Considered under write-triggers; rejected in favor of the simpler per-answer model. Phase 11's review queue will handle due-based selection.

</deferred>
