### Anti-Patterns Found

No blocking anti-patterns detected.

Notable observations (non-blocking):
- `saveSessionResults` also performs a mastery upsert at session end as a batch fallback (lines 154–260). This is intentional (documented in comments) and supplementary to `recordVocabAnswer`'s per-answer writes. The dual-write path could double-count reps for a session, but the session-end path uses `seenVocabIds` deduplication, limiting it to one upsert per unique vocab per session. Acceptable at current phase.
- `recordVocabAnswer` uses a raw `sql\`\`` template for the SELECT in step 2 (line 315) rather than parameterized Drizzle where clauses. This is consistent with the pattern used elsewhere in the file and is not a security issue since `userId` and `vocabItemId` come from server-side inputs only.
- The `reveal-reading` button hides when `targetTier === 1` (cold-start / new word). This is correct UX (full help already visible), but it means a user on their first encounter never sees the button. This is intentional per the phase spec.
