Wire the dead Phase 7 tables. Add a per-answer server action that writes one
user_exercise_log row and upserts user_vocab_mastery via FSRS, plus two read
endpoints: a batch tier-loader the UI hits at session start and a single-word
mastery-detail endpoint the feedback panel hits on tap-to-inspect.

"new → learning → review" progression actually moves. No batching at the session
boundary — the user-visible semantic is "persisted on answer" (CONTEXT lock).

Output: One new server action + two new route handlers, all using the Plan 01
FSRS primitives. saveSessionResults is left in place for song-level progress;
this plan adds vocab-level persistence alongside it.

    ```ts
    import { userVocabMastery, userExerciseLog } from "@/lib/db/schema";
    import { ratingFor } from "@/lib/fsrs/rating";
    import { scheduleReview } from "@/lib/fsrs/scheduler";
    import type { ExerciseType } from "@/lib/exercises/generator";

    interface RecordAnswerInput {
      // TODO: replace with Clerk userId from auth()
      userId: string;
      vocabItemId: string;            // target word's UUID; NEVER pass distractor IDs
      songVersionId: string | null;   // null for kana-only exercises (Phase 9)
      exerciseType: ExerciseType;
      correct: boolean;
      revealedReading?: boolean;      // true if user tapped the reveal-reading hatch
      responseTimeMs: number;
    }

    interface RecordAnswerResult {
      newTier: 1 | 2 | 3;
      newState: 0 | 1 | 2 | 3;
      reps: number;
      lapses: number;
      due: string;                    // ISO timestamp
    }

    export async function recordVocabAnswer(input: RecordAnswerInput): Promise<RecordAnswerResult>
    ```

    Implementation, in one db.transaction(...) block:
      1. Compute `rating = ratingFor(input.exerciseType, input.correct, { revealedReading: input.revealedReading })`.
      2. SELECT existing user_vocab_mastery row WHERE user_id = userId AND vocab_item_id = vocabItemId
         (FOR UPDATE not strictly required at MVP — single-user contention is low,
         but use a simple SELECT + onConflictDoUpdate pattern below).
      3. Build a `MasteryRow | null` from the SELECT result (null if no row).
         The drizzle row already includes `intensity_preset` from the schema and
         maps cleanly to Plan 01's `MasteryRow` type (which now exposes
         `intensity_preset?: IntensityPreset`).
      4. `const next = scheduleReview(prev, rating, prev?.intensity_preset ?? "normal")`
         — typechecks against Plan 01's updated MasteryRow.
      5. UPSERT user_vocab_mastery via `db.insert(...).onConflictDoUpdate({ target: [user_id, vocab_item_id], set: {...} })`.
         Set: stability, difficulty, elapsed_days, scheduled_days, reps, lapses,
         state, due, last_review, updated_at = NOW(). Insert values must include
         intensity_preset = "normal" (default) plus the fields above.
      6. INSERT user_exercise_log with vocab_item_id, song_version_id (nullable),
         exercise_type, rating, response_time_ms = input.responseTimeMs.
         CONTEXT: "one row per question attempt". One row per call to this action.
         Distractors are never persisted — caller must only pass the target's vocabItemId.
      7. Import tierFor and return { newTier: tierFor(next.state), newState, reps, lapses, due: next.due.toISOString() }.

    Add a JSDoc note at the top of recordVocabAnswer:
      "Per-CONTEXT decision: distractor words receive NO mastery or log row.
       The caller MUST pass only the target vocabItemId of the question being asked."

    Error handling: catch DB errors and rethrow with context. If lessons happen
    to lack a vocab_item_id (legacy data), the action is not called by the UI
    (Plan 03 guards on Question.vocabItemId being a UUID); but defensively, this
    action throws if vocabItemId is empty.
