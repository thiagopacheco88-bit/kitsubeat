  <action>
    Create `src/lib/fsrs/rating.ts`:

    1. Export `FSRSRating = 1 | 2 | 3 | 4` — alias for ts-fsrs Rating values
       (1=Again, 2=Hard, 3=Good, 4=Easy). Re-export ts-fsrs `Rating` enum
       too for callers that prefer the named form.
    2. Export `RATING_WEIGHTS: Record<ExerciseType, FSRSRating>` — the rating
       awarded for a CORRECT answer per exercise type. CONTEXT decision:
       production-flavored types weigh more than pure-surface types. Use:
         vocab_meaning:  3 (Good)   — recognition
         meaning_vocab:  4 (Easy)   — production-flavored, hardest direction
         reading_match:  2 (Hard)   — pure surface, easiest, weighed lower
         fill_lyric:     3 (Good)   — recognition + context
       Document in a JSDoc block at the top of the constant that these are
       tunable defaults locked here so future tuning happens in ONE place.
    3. Export `ratingFor(exerciseType: ExerciseType, correct: boolean, opts?: { revealedReading?: boolean }): FSRSRating`:
       - If `opts.revealedReading === true`, return 1 (Again) regardless of correct flag
         — CONTEXT: "the reveal counts as a non-correct answer for FSRS purposes…
         treat it as equivalent to a wrong answer or 'Again' rating".
       - Else if `correct === false`, return 1 (Again).
       - Else return RATING_WEIGHTS[exerciseType].
    4. Import `ExerciseType` from `@/lib/exercises/generator` to keep one source of truth.

    Create `src/lib/fsrs/__tests__/rating.test.ts` (RED first):
    - All 4 exercise types × {correct, wrong} = 8 deterministic cases.
    - Wrong answers all return 1.
    - Correct meaning_vocab returns 4; correct reading_match returns 2.
    - Correct meaning_vocab > correct reading_match (the weight ordering invariant).
    - revealedReading: true with correct: true returns 1 (hatch always penalizes).
    - revealedReading: true with correct: false returns 1.

    Implementation must NOT import drizzle, react, or any DB module. Pure TS.
  </action>
  <verify>npx vitest run src/lib/fsrs/__tests__/rating.test.ts — all cases green.</verify>

        0 (New) → 1
        1 (Learning) → 2
        2 (Review) → 3
        3 (Relearning) → 2   ← CONTEXT: "relearning treated as learning for tier purposes"
        any other (defensive) → 1
    - JSDoc references the locked CONTEXT decision. NO stability thresholds —
      pure state-driven mapping per CONTEXT: "Stability thresholds are not used
      directly — FSRS's own state transitions drive the tier."

    Create `src/lib/fsrs/__tests__/tier.test.ts` (RED first):
    - Each of the 4 states maps to its expected tier (4 cases).
    - Defensive case: tierFor(99) returns 1.
    - Pure-kana note: tierFor does not branch on word shape (test that calling
      tierFor(0) returns 1 regardless — kana words just render the kana itself
      at every tier; this is a renderer concern, not a tier concern).

    Create `src/lib/fsrs/scheduler.ts`:
    - Import `fsrs`, `Rating`, `State`, `createEmptyCard` from ts-fsrs.
    - Import `INTENSITY_PRESETS` from `@/lib/fsrs-presets`.
    - Export type `ScheduledUpdate`:
        { stability: number; difficulty: number;
          elapsed_days: number; scheduled_days: number;
          reps: number; lapses: number;
          state: 0 | 1 | 2 | 3;
          due: Date; last_review: Date; }
    - Export type `MasteryRow` mirroring the user_vocab_mastery scalar columns
      that scheduleReview reads from (stability/difficulty/elapsed_days/
      scheduled_days/reps/lapses/state/due/last_review). All nullable except
      the integer counters & state. Also include
      `intensity_preset?: IntensityPreset` (optional) for callers that select the
      whole row from drizzle and pass it through to scheduleReview. JSDoc note:
      `intensity_preset` is carried on the row for UI/DB use only — the scheduler
      itself ignores it (the active preset is passed as the explicit `intensity`
      argument). Import `IntensityPreset` from `@/lib/fsrs-presets`.
    - Export `scheduleReview(prev: MasteryRow | null, rating: FSRSRating, intensity: IntensityPreset = "normal", now: Date = new Date()): ScheduledUpdate`:
        1. Build an FSRS card from prev (or createEmptyCard(now) when prev is null).
           When prev is non-null, hydrate from its scalar columns: due, stability,
           difficulty, elapsed_days, scheduled_days, reps, lapses, state, last_review.
        2. Instantiate `fsrs(INTENSITY_PRESETS[intensity])`.
        3. Call `f.next(card, now, rating).card` to get the next card.
        4. Map the card back to ScheduledUpdate scalars (state values 0/1/2/3
           must match the smallint encoding used in user_vocab_mastery — verify
           against ts-fsrs State enum: New=0, Learning=1, Review=2, Relearning=3).

    - Lapse: starting from a state=2 (Review) row, scheduleReview(prev, 1 /* Again */)

    - Easy promotion: from state=1 (Learning), scheduleReview(prev, 4 /* Easy */) eventually
      reaches state=2 (Review) — single-step verification: result state is 1 or 2,
      reps incremented.
    - Intensity respected: scheduleReview(null, 3, "intensive") and ("light") both succeed
      and return ScheduledUpdate; the "intensive" `due` is no further out than the
      "light" `due` for the same starting card.
    - Pure (no DB): test runs without any drizzle or DB module loaded.
